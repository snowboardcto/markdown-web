# Story 2.4: Media embedding

Status: review

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As an author,
I want images and video to render inline,
so that my pages are rich, not text-only.

## Acceptance Criteria

1. **Given** a content page containing a relative image embed — the epic example `![](media/powder.jpg)` on `content/x.md` (page `/x`), with the asset present in the vault at `content/media/powder.jpg` — **When** the page is built and viewed **Then** the rendered HTML contains an `<img>` whose `src` resolves to a **served URL** (a root-absolute `/media/powder.jpg`, NOT the literal relative `media/powder.jpg` which would 404 against the page route `/x`), the asset is actually present in the build output and served (HTTP 200, correct image `Content-Type`), and the image displays inline (a non-zero rendered size in the browser). The rewrite happens at **build time** and the emitted HTML is plain static `<img src="/…">` — no `client:*` island, no runtime JS (FR-5/FR-7 preserved). *(AC1 — relative image embed renders inline AND its asset is served from the vault path; FR-3, EXPERIENCE.md line 77 "working links and inline media")*

2. **Given** relative media `src` values, resolved against the **current page's own route/dir** exactly as Story 2.3 resolves relative `.md` links (the SAME `pathToSlug`/`path.posix` resolution rules, in slug space), **When** the page renders **Then**: a same-dir embed `![](media/powder.jpg)` from `/x` → `src="/media/powder.jpg"`; a nested-page embed resolves against the page's directory — from a page at `/sub/page`, `![](../media/powder.jpg)` → `/media/powder.jpg` and `![](pic.png)` → `/sub/pic.png`; a leading `./` is collapsed (`./media/powder.jpg` from `/x` → `/media/powder.jpg`); spaces/percent-encoding in the path are handled deterministically and decoded **per segment** so an encoded separator cannot manufacture a new path part (mirroring 2.3's F1 guard) — and the asset is copied to / served at the **same** resolved path. **Media paths are NOT github-slugged** — unlike `.md`→route slugs, an asset filename must be preserved **byte-for-byte** (case and all) so `Powder Day.JPG` stays `Powder%20Day.JPG`/the real on-disk name and the served file matches the `src` exactly (a slugged `src` would point at a file that does not exist). Directory segments are likewise preserved verbatim. *(AC2 — relative media resolution honours nested dirs / `..` / `./` / encoded chars, resolved against the current page's location, with asset filenames preserved exactly; FR-3)*

3. **Given** the same render path, **non-relative** and **non-media** references are left exactly as authored **When** the page renders **Then** an absolute external media URL (`![](https://example.com/a.png)`, a `data:` URI, a protocol-relative `//host/a.png`, or a root-absolute `/already/served.png`) keeps its `src` unchanged (external/already-served assets are not rewritten and not copied), and the rewrite applies to media elements only — `<img>`, `<video>`, `<audio>`, and their nested `<source>`/`poster` — leaving ordinary `<a href>` links to the Story 2.3 plugin (this story does **not** re-handle `.md` link rewriting). Only *relative* `src`/`poster` values are rewritten. *(AC3 — only relative media src/poster is rewritten; external/data/absolute pass through; `<a>` links remain 2.3's concern; FR-3, scope boundary vs 2.3)*

4. **Given** an image embed with author-provided alt text — `![a skier in deep powder](media/powder.jpg)` — **When** the page renders **Then** the `<img>` carries `alt="a skier in deep powder"` (alt text passes through untouched; the src rewrite must not drop or alter `alt`, `title`, `width`, `height`, or any other authored attribute), and an empty-alt embed `![](media/powder.jpg)` emits `alt=""` (decorative — the standard markdown→HTML behaviour, preserved). Accessibility floor (NFR-6 / UX-DR9): alt text is the author's accessibility affordance and must survive the rewrite. *(AC4 — alt/title/other attributes pass through unchanged; FR-3, NFR-6, UX-DR9)*

5. **Given** non-image media — a video embed authored as raw HTML in markdown (e.g. `<video src="media/clip.mp4" controls></video>` or `<video controls><source src="media/clip.mp4" type="video/mp4"></video>`), since GFM markdown has no native video syntax — **When** the page renders **Then** the relative `src`/`poster` on `<video>`/`<audio>`/`<source>` is rewritten to its served `/…` path the same way images are, the media file is copied into the build output, and it is served with the correct `Content-Type` (e.g. `video/mp4`). Per EXPERIENCE.md, the WEB path serves standard HTML `<video>`/`<img>`; native-client video via MediaElement is Epic 3's concern and is **out of scope** here. *(AC5 — relative `<video>`/`<audio>`/`<source>`/`poster` assets are served too, not just images; FR-3; scope: web only, not native MediaElement)*

6. **Given** a relative media embed whose asset does **not** exist in the vault — `![](media/missing.jpg)` — **When** the page builds **Then** the build does **not crash** and does **not silently exit-0 with a broken page**: the `<img>` still rewrites to its would-be served path (`/media/missing.jpg`) and, because no such file is copied, the browser/crawler gets a normal **404 for that asset** (a broken-image state, never a build crash or a thrown exception) — consistent with Story 2.3's "missing target → clear not-found, never a crash" philosophy. The decision on whether to additionally **fail the build loudly** on a referenced-but-missing asset is a Dev decision documented in the Dev Agent Record (see "Missing-asset policy" in Dev Notes); the floor is *never crash mid-render and never emit a malformed page*. *(AC6 — missing asset → broken-image/404, never a build crash; FR-3, mirrors 2.3 AC4 robustness)*

7. **Given** the media-embedding changes are purely additive to the Story 2.1/2.2/2.3 render path **When** the site builds and the full Playwright suite runs **Then** the build still exits 0 and emits one page per `content/**/*.md`, every page stays server-rendered / JS-free / semantically well-formed (single `<html lang>`→`<head>`→`<body>`→`<main>`/`<article>`, one `<h1>`, doctype), the GitHub theme is unchanged, Story 2.3's link rewriting + custom 404 still work, and **all 66 existing Playwright specs (`web/tests/*.spec.ts`) still pass unchanged** alongside the new media specs. The asset-copy step must not change route emission, slug collision detection, or the content collection. *(AC7 — no regression to 2.1 crawlable/JS-free, 2.2 theme, or 2.3 linking/404; FR-5, FR-7, NFR-3)*

### The load-bearing decisions

**Decision 1 — rehype `<img>/<video>` `src` rewrite (NOT remark, NOT `astro:assets`).** At the rehype (HAST) stage, markdown image embeds are already `<img src="…">` elements; rewriting `properties.src` is a one-line mutation, exactly parallel to how 2.3 rewrites `<a href>`. A remark (MDAST) plugin would have to handle `image` and raw-HTML `<video>` nodes separately. `astro:assets`/`<Image>` is the WRONG tool: it is an image-OPTIMISATION pipeline (resize, format-convert, hashing) — the AC is "the asset is served from the vault path", i.e. a faithful verbatim copy, not an optimised/renamed variant. Architecture (line 104) says "resolve relative paths against the vault", and 2.3's precedent is a build-time rehype `href` rewrite — the media `src` rewrite is the exact analogue. [Source: architecture.md line 104; web/src/lib/rehype-md-links.mjs; Astro assets docs (out of scope)]

**Decision 2 — serve vault media by build-time copy into `dist` (NOT Astro `public/`, NOT symlink).** SWA serves from `web/dist` (deploy-web.yml line 59). The cleanest way to get `content/media/**` into `dist/media/**` without violating "`content/` is the single source of truth … No content in code" (architecture.md line 143) is an `astro:build:done` integration hook that copies non-`.md` vault files into the just-written `dist`. Putting media under `web/public/` would duplicate vault content into `web/` (architecture violation + desync risk); a symlink is fragile across CI/OS and Astro may not follow it. The copy is deterministic, keeps the vault canonical, and matches 2.3's "choose the simplest robust static approach" instinct. **Path invariant:** asset path relative to `content/`, verbatim, root-absolutised → both the copied `dist` path and the rewritten `src` agree. [Source: deploy-web.yml line 59; architecture.md line 143; Astro `astro:build:done` hook]

**Decision 3 — assets are NOT slugged; only the page-relative DIRECTORY is resolved.** This is the single most important divergence from 2.3. A `.md` link target gets `pathToSlug`'d (lower-cased, github-slugged) because it must match an emitted ROUTE. An asset must NOT be slugged: the served file is the literal on-disk file, so `src` must point at the real filename. Resolve `./`/`..`/nested-dir against the page dir (in POSIX space) but emit the asset's own segments verbatim. A slugged asset `src` would 404 (no file at the slugged name). [Source: AC2; web/src/lib/slug.mjs intended use]

### The missing-asset / broken-image state (AC6)

Mirrors 2.3's "missing target → clear not-found, never a crash": the plugin rewrites `media/missing.jpg` → `/media/missing.jpg` without checking existence; the copy step copies only files that exist, so the browser gets a normal 404 for the asset (a broken-image placeholder), never a build crash. **Floor:** neither the rewrite nor the copy may throw on a missing reference. Recommended (not required): a build-time `console.warn` listing referenced-but-missing assets, mirroring `rehype-md-links.mjs`'s `unresolvedPages` warn — so a typo is visible, not silent. Do NOT hard-fail the build by default. [Source: AC6; web/src/lib/rehype-md-links.mjs lines 84–88]

**Two non-negotiable floors, two failure points:**
1. **Rewrite floor (rehype, per-page):** the `src` rewrite NEVER checks existence and NEVER throws — a missing asset rewrites to its would-be served path exactly like a present one. The rewrite cannot know what the copy step will/won't copy (rehype runs per-page; the copy is once-per-build, possibly after) so it must be existence-agnostic. A malformed `%`-escape or `%2F`-smuggle degrades to UNREWRITTEN (rows 12–14), which is a SEPARATE concern from "asset missing" — do not conflate.
2. **Copy floor (`astro:build:done`):** copies only files that EXIST in `content/`; a referenced-but-missing asset simply has no file → 404. The copy walks `content/` (what exists), NOT the set of referenced `src` (what's wanted), so a missing reference is structurally a no-op for the copy, never a lookup-that-throws.

**If implementing the recommended warn (the visibility nicety):** the rehype plugin may collect the SET of relative asset paths it rewrote (module-level `Set`, deduped, like `unresolvedPages`); the `astro:build:done` hook (which has `content/`'s real file list) can diff `referenced − present` and `console.warn` the missing ones ONCE (deduped across pages). This crosses the rehype↔integration boundary — keep it OPTIONAL and behind the floor: if the wiring is awkward, ship the floor (never-crash, 404) WITHOUT the warn and document the deferral. The warn must itself never throw (an empty/again-missing set is a clean no-op). The choice (warn vs no-warn vs fail) is recorded in the Dev Agent Record per AC6. [Source: AC6; web/src/lib/rehype-md-links.mjs lines 54, 84–88 `unresolvedPages` warn precedent]
