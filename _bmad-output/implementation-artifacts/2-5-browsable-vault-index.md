# Story 2.5: Browsable vault index

Status: review

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a reader,
I want an entry surface that lists the vault's pages,
so that I can find my way without typing URLs.

## Acceptance Criteria

1. **Given** a vault of several `.md` pages (the current `content/**/*.md` set), **When** I visit the index surface, **Then** the page **lists or links EVERY** vault page — exactly one link per emitted content route (the same set `[...slug].astro` builds: `/x`, `/gear-guide`, `/my-notes`, `/no-h1`, `/empty`, `/readme`, `/sub`, `/sub/page`, `/sub/page2`, `/sub/sibling`, `/my-notes-dir/page`, … whatever `content/**/*.md` currently yields), with no vault page omitted and no link to a non-page. **Completeness invariant:** the count of index `<a>` links to content routes EQUALS the number of `getCollection('pages')` entries (one-to-one, no omissions, no extras, no self-link to `/`). **Degenerate cases:** an **empty vault** (zero `.md`) yields a valid index page that renders the shell + `<h1>` and an explicit empty-state (an empty `<ul>` or a "no pages yet" line) and does NOT crash the build; a **single-page vault** yields exactly one `<li>`. The listing is **generated from the content collection** (`getCollection('pages')`), NOT a hand-maintained list, so adding a new `.md` to the vault and rebuilding makes it appear automatically (the "add a file, get a page" promise, EXPERIENCE.md line 77 — extended to "and it shows up in the index"). *(AC1 — the index lists/links every vault page one-to-one (incl. empty/single-page degenerate cases), generated from the collection; FR-4, epics.md Story 2.5)*

2. **Given** each listed page, **When** the index renders, **Then** every link `href` is a **resolved site route** (root-absolute `/` + `entry.id` — the authoritative route slug the catch-all emits; never a literal `*.md` path and never a dead route), so clicking any entry navigates to a real 200 page, and **the link text is a human-readable label** derived by the SAME title precedence the destination page uses (front-matter `title` → first `# H1` → slug-derived Title Case). **Anti-drift is MANDATORY, not optional:** `slugToTitle` currently lives INLINE in `[...slug].astro` (lines 72–78) — it MUST be extracted to a shared `web/src/lib/` helper and imported by BOTH `[...slug].astro` and `index.astro` (a behaviour-preserving extraction; the 94 specs stay green), so the index label and the destination `<title>`/`<h1>` cannot diverge from copy-paste rot. **Resolve the H1 divergence explicitly (Decision D):** if the index does NOT `render()` each entry (the cheap path: `entry.data.title || slugToTitle(entry.id)`), then for any page whose title comes ONLY from its H1 (no front-matter `title`), the index label is the slug-derived Title Case while the destination `<title>` is the H1 — a KNOWN, documented divergence (e.g. `content/no-h1.md`, and any H1-only page). Either eliminate it by `render()`-ing per entry (label EXACTLY matches destination), or document it and have the test assert the actual chosen behaviour (not an idealised one). *(AC2 — every index link resolves to a live route with a readable title label; title logic is a SHARED extracted helper; H1-divergence resolved explicitly; FR-2/FR-4)*

3. **Given** the index is the vault's entry surface, **When** I navigate the site, **Then** I can **reach any vault page by navigation alone** — the index is itself reachable at a stable, obvious URL (the site root `/` is the entry surface; see Decision A) and links to every page, and existing inter-file links (Story 2.3) plus the 404's "Go back home" `<a href="/">` all lead to a surface from which any page is reachable. No page is an orphan that requires typing its URL. *(AC3 — reachability: the index is the navigable entry point to all pages; FR-4 "reach any page by navigation alone")*

4. **Given** the vault has nested pages (`content/sub/*.md`, `content/My Notes Dir/*.md`), **When** the index lists them, **Then** nested pages are **included and presented coherently** — either grouped by their top-level directory (a small section/`<ul>` per dir) OR a single flat alphabetical list of all routes; pick ONE deterministic ordering and document it (Decision B). A `content/<dir>/index.md` (which routes to `/<dir>`, e.g. `sub/index.md` → id `sub` → `/sub`) is listed once at its collapsed route, not duplicated as both `/sub` and a phantom `/sub/index`. The presentation is **stable/deterministic AND CI-stable** across rebuilds: sort EXPLICITLY by `entry.id` (or by label), and the sort MUST be locale-independent so it cannot reorder between a dev machine and CI — use `a.id.localeCompare(b.id, 'en')` (or a plain code-unit comparison via `< / >`), NOT a default `localeCompare()` whose ordering depends on the host ICU/locale, and NOT `getCollection`'s iteration order (not a stable contract). The chosen key produces a single, reproducible ordering documented in Decision B; the test pins that exact order. *(AC4 — nested pages included, CI-deterministic ordering, no index-collapse duplication; FR-4)*

5. **Given** the Story-1.1 placeholder `web/src/pages/index.astro` ("Coming soon") currently owns `/`, **When** this story ships the real index at `/`, **Then** the placeholder is **retired/replaced** (no "Coming soon" page remains at `/`), the new `/` renders through the shared `Page` layout (Story 2.1 shell + Story 2.2 GitHub theme), and the route-collision rules still hold: `/` is owned by `index.astro` (the index page), while `content/**/*.md` pages stay owned by `[...slug].astro`. **Correction — verified against `slug.mjs`:** a root `content/index.md` does NOT route to `/index`; `pathToSlug` collapses a bare `index` to `''` (slug.mjs line 32, `.replace(/^index$/, '')`), so its `entry.id` is `''` and it routes to `/` — which would **collide** with `index.astro`, not coexist. (The `[...slug].astro` header comment claiming `/index` is stale relative to the actual slug logic; trust the code.) This story does NOT add a `content/index.md`. The index reads the collection and MUST therefore guard the empty-`id` case: an entry whose `id === ''` is NOT listed as a phantom `/`-link in its own listing (it would be the `/` route itself); none exists today, and Decision C documents the interaction. *(AC5 — the 1.1 placeholder is replaced by the real index at `/`, themed via the shared layout, no route-ownership regression; a root `content/index.md` would collide at `/`, not route to `/index`; epics.md Story 1.1 context)*

6. **Given** the index must match the established reading experience, **When** it renders, **Then** it is **themed (Story 2.2 `github.css`), server-rendered, JS-free, and crawlable** — built-time static HTML through the `Page` layout (single `<html lang>` → `<head>` → `<body>` → `<main>`/`<article>`, exactly one `<h1>`, valid doctype, WCAG-AA contrast inherited), with **no `client:*` island and no runtime JS** (the listing is plain `<a>` links a crawler can follow, FR-5/FR-7/NFR-4). The index carries **no site-header / pitch-card / get-client chrome** (that is Story 2.6, which later layers chrome onto ALL pages including this one) — this story ships the **listing surface ONLY**. *(AC6 — themed, JS-free, crawlable, semantic shell; listing-only scope, no 2.6 chrome; FR-5/FR-7, EXPERIENCE.md)*

7. **Given** the index is purely additive to the Stories 2.1–2.4 render path, **When** the site builds and the full Playwright suite runs, **Then** the build still exits 0, still emits one page per `content/**/*.md` (the index does NOT add, drop, or re-slug any content route, does not touch the content collection, slug logic, link/media rehype plugins, or 404), the GitHub theme + 2.3 linking/404 + 2.4 media all still work, and **all 94 existing Playwright specs (`web/tests/*.spec.ts`) still pass unchanged** alongside the new index specs. `astro check` stays at 0 errors. *(AC7 — no regression to 2.1 render / 2.2 theme / 2.3 linking+404 / 2.4 media; the 94-spec baseline stays green; FR-5/FR-7, NFR-3)*

### The load-bearing decisions

**Decision D — label source: reuse the `[...slug].astro` title precedence via a SHARED EXTRACTED helper (mandatory).** `slugToTitle` lives INLINE in `[...slug].astro` (lines 72–78) today; EXTRACT it to `web/src/lib/` and import from both files (behaviour-preserving; 94 specs stay green).
