## Story

As a reader,
I want to choose which AI personality renders pages,
so that I control how content meets me.

**IN (the selector + re-render-in-place + key UX):** the `PersonaRegistry.Seed` (Agent — the ordered seed list the selector enumerates; Basic first + pass-through, the rest placeholder-prompt + `IsPassThrough=false`); the toolbar `PersonalitySelector` `ComboBox` (App UI — labeled, keyboard-reachable, default Basic); the `PersonalitySelectionViewModel`/`IPersonalitySelection` selection state (App — drives the gateway, replacing the constant `() => Persona.Basic`); the `PersonalityRerenderCoordinator` re-render-in-place path (App — holds the RAW current markdown, re-personalizes WITHOUT re-fetch, last-wins); the `ApiKeyPromptViewModel` + `ApiKeyEntryDialog` key-entry UX (App — stores via `ISecretStore.SetApiKey`, discloses BYO-key, never crashes on blank/missing); the state/notice surfacing (App — transforming affordance + `LastNotice`/`LastOutcome`, Basic instant). ### Re-render-in-place coordinator (the no-re-fetch + last-wins mechanism)

The 4-1 seam runs the gateway INSIDE the fetch path (`FetchEndpointAsync`, `MainWindow.xaml.cs` 78–93): a NAVIGATION fetches then personalizes. A personality SWITCH is **NOT a navigation** — no history push, no fetch. 4.2 adds a SEPARATE re-render path:

- **Hold the RAW markdown.** In `FetchEndpointAsync`, after a successful fetch and BEFORE `_gateway.ResolveMarkdownAsync`, capture `fetched.Markdown` (the RAW source) + `pageUrl` into the coordinator (`_rerender.SetCurrentPage(fetched.Markdown, pageUrl)`). The navigation continues to render the resolved markdown exactly as 4-1. Holding the RAW (not the resolved/transformed) markdown is essential: switching Basic→Cozy→Basic must return to the byte-identical original, so the engine always transforms FROM SOURCE, never from a prior transform.
- **Re-render on switch.** `_selection.SelectionChanged` → `await _rerender.RerenderAsync()`. `RerenderAsync` re-runs the HELD raw markdown through `_gateway.ResolveMarkdownAsync(heldRaw, heldUrl, ct)` (the gateway reads the now-current persona) and calls the render-sink (`(md, url) => _contentHost.ShowMarkdown(md, url)`) — the SAME sink the controller uses. **No `MarkdownFetcher`/`FetchEndpointAsync` call — zero GETs.**
- **Last-wins.** `RerenderAsync` claims `++_generation` before the await and re-checks `myGen == _generation` after the gateway resolves; a superseded re-render (a newer switch, OR a navigation that has since rendered) is dropped (no `ShowMarkdown`). This mirrors `NavigationController.RunAsync` (lines 152–156). A switch mid-navigation: the navigation's render and the switch's re-render each re-check their generation; the last to claim wins. (Consider sharing a single generation counter between the controller and the coordinator OR keeping them independent and letting the held-markdown update + the re-check serialize on the STA thread — decide & document; the simplest correct option on the single-threaded STA UI is the coordinator's own generation + the STA single-thread ordering, since both the navigation render-sink and the switch re-render run on the UI thread.)
- **Nothing held yet → no-op.** A switch before any page loads has no held markdown → `RerenderAsync` returns without rendering (no fetch, no crash). First-run default is Basic, so even a pre-load switch is harmless.

The coordinator is factored so the no-re-fetch + last-wins logic is `[Fact]`-testable WITHOUT a window: inject a `PersonalizationGateway` (over a fake `ILlmClient`) + a recording `Action<string,Uri>` render-sink; assert the sink input + that no fetch delegate was invoked (the coordinator holds no fetcher at all — zero-re-fetch by construction; a counting fake fetcher in the test asserts 0 if the test wires one to the navigation path).

### Key-entry UX (BYO-key — the testable invariants)

- **Stores via the SAME `ISecretStore` the engine reads.** `ApiKeyPromptViewModel(ISecretStore)` → `TrySave` → `SetApiKey(trimmed)`. The composed `MainWindow` passes its single `_secretStore` (the `DpapiSecretStore`) to both the engine (via `AnthropicLlmClient`) and the prompt VM, so a saved key is immediately visible to the next re-render's engine call.
- **Discloses BYO-key (D1).** `DisclosureText` is plain, key-free: the key is stored locally + encrypted (DPAPI, per-user) and sent only to the reader's chosen provider over TLS — never to a Markdown-Web server. Reuse the 4-1 D1 framing.
- **Blank/missing → clear state, never crash.** `TrySave("")`/`"   "` does NOT call `SetApiKey`, returns false, no throw; the page stays the original/basic render with the `NeedsKey` notice. The reader can re-open the prompt and try again.
- **Never logs the key.** The VM/dialog never writes the key to a notice/log/exception. The no-leak `[Fact]` asserts a test key appears in no surfaced coordinator string (reuse the 4-1 no-leak discipline).
- **Trigger.** The re-render's `LastOutcome == NeedsKey` is the trigger to surface the affordance. In CI the dialog is constructed (not Shown); the trigger logic is `[Fact]`-tested via the coordinator's `LastOutcome` (no window needed).

### State / notice patterns (AC5 — reuse 4-1's notice plumbing)

The gateway already exposes `LastNotice`/`LastOutcome` (4-1). The coordinator passes them through. The UI:
- **Basic = instant pass-through:** `PassThrough` outcome, null notice, no LLM call, byte-identical — no transforming affordance.
- **Non-Basic transform:** a non-blocking transforming affordance while the gateway resolves (the prior `FlowDocument` stays shown; the re-render is `await`ed; last-wins drops a superseded one). On success: `Transformed`, null notice. On failure: `FellBack`, the held original rendered, a key-free notice. On no key: `NeedsKey`, the held original rendered, a key-free notice → the key-entry trigger (AC4).
- The notice surface is a small App-side text (a toolbar/status `TextBlock` near the selector). Non-blocking; never a modal that freezes reading.
