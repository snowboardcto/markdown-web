## Story

As a reader,
I want to choose which AI personality renders pages,
so that I control how content meets me.

## Context note (this is the SECOND Epic-4 story — the SELECTOR + re-render-in-place + key UX)

Story 4.1 shipped the **Epic-4 agent foundation** (Status: done, CI run #42 green): the `Agent` module (`ILlmClient`/`AnthropicLlmClient`, `ISecretStore`/`DpapiSecretStore`, `PersonalityEngine`, `Persona`/`Persona.Basic`, `ReaderContext`, `PersonalizationResult`/`PersonalizationOutcome`), BYO-key DPAPI storage with graceful totality, the no-server-side-rewrite (NFR-5) proof, and the App render-time seam: an `App.PersonalizationGateway(PersonalityEngine engine, Func<Persona> selectedPersona)` whose `ResolveMarkdownAsync(string fetchedMarkdown, Uri pageUrl, CancellationToken ct)` runs the fetched markdown through the engine and returns `result.Markdown`. **The gateway's persona selector is CURRENTLY the constant `() => Persona.Basic`** (`MainWindow.xaml.cs` line 45), and the gateway runs **inside the fetch path** (`FetchEndpointAsync`, `MainWindow.xaml.cs` lines 78–93). The toolbar (3.2) **reserved a slot** to the right of Back/Forward/Reload/AddressBar for "the Epic-4 personality selector" (`MainWindow.xaml` lines 18–19: *"Reserved space for the Story 3.2 address bar (and the Epic 4 personality selector)."*).

**Story 4.2 builds the personality SELECTOR mechanism + re-render-in-place + the API-key entry UX.** It is the second Epic-4 story:

**Q-Token (the re-render generation token) — RESOLVED: an INDEPENDENT coordinator generation + STA single-thread ordering. NOT shared with `NavigationController._generation`.**

- **Decision.** `PersonalityRerenderCoordinator` owns its OWN private `int _generation` (mirroring `NavigationController._generation` / `AddressBarViewModel._submitGeneration`). It is NOT the same counter as `NavigationController`'s. `RerenderAsync` claims `int myGen = ++_generation` before the awaited `gateway.ResolveMarkdownAsync`, and after the await re-checks `myGen == _generation`; if not, it returns WITHOUT calling the render-sink (a superseded re-render is dropped).
- **Justification.** (1) A switch is NOT a navigation — it has no history push, no fetch, no broken-sink path; folding it into `NavigationController._generation` would force `NavigationController` internals to change (the story forbids that — `NavigationController` is UNCHANGED). (2) Both the navigation render-sink call and the switch re-render run on the SINGLE STA UI thread; there is no cross-thread race to coordinate, so two independent monotonic counters + the STA serialization are sufficient and strictly simpler than a shared one. (3) Independence keeps the no-regression guarantee airtight: 4.2 adds NO edit to `NavigationController`.
- **Precedence — how nav×switch never renders stale (the load-bearing interaction, PINNED):**
  - **A fresh navigation supersedes a pending re-render.** When a navigation completes, its render-sink call `SetCurrentPage(newRawMarkdown, newUrl)` updates the held RAW markdown (see the wiring in Task 3) AND `++_generation` on the coordinator is bumped (the coordinator subscribes to "a navigation rendered" by having `SetCurrentPage` bump its generation).   - **A fresh re-render supersedes a pending re-render.** Two rapid switches: the second claims a higher `_generation`; the first resumes stale and is dropped. **Last switch wins.**
  - **A switch mid-navigation.** The switch's `RerenderAsync` re-runs the CURRENTLY-held RAW markdown; if a navigation is in flight, the held markdown is still the OLD page's (the nav hasn't called `SetCurrentPage` yet) — but the moment the nav lands it calls `SetCurrentPage` (bumping `_generation`), so any switch re-render still in flight is dropped, and the nav renders the fresh page. ### No-re-fetch proof mechanism (the load-bearing AC2) — PINNED

The EXACT chain that holds the RAW markdown and reuses it with ZERO fetcher involvement:

### Re-render-in-place coordinator (the no-re-fetch + last-wins mechanism)

The 4-1 seam runs the gateway INSIDE the fetch path (`FetchEndpointAsync`, `MainWindow.xaml.cs` 78–93): a NAVIGATION fetches then personalizes. A personality SWITCH is **NOT a navigation** — no history push, no fetch. 4.2 adds a SEPARATE re-render path:

- **Hold the RAW markdown.** In `FetchEndpointAsync`, after a successful fetch and BEFORE `_gateway.ResolveMarkdownAsync`, capture `fetched.Markdown` (the RAW source) + `pageUrl` into the coordinator (`_rerender.SetCurrentPage(fetched.Markdown, pageUrl)`). The navigation continues to render the resolved markdown exactly as 4-1. Holding the RAW (not the resolved/transformed) markdown is essential: switching Basic→Cozy→Basic must return to the byte-identical original, so the engine always transforms FROM SOURCE, never from a prior transform.
- **Re-render on switch.** `_selection.SelectionChanged` → `await _rerender.RerenderAsync()`. `RerenderAsync` re-runs the HELD raw markdown through `_gateway.ResolveMarkdownAsync(heldRaw, heldUrl, ct)` (the gateway reads the now-current persona) and calls the render-sink (`(md, url) => _contentHost.ShowMarkdown(md, url)`) — the SAME sink the controller uses. **No `MarkdownFetcher`/`FetchEndpointAsync` call — zero GETs.**
- **Last-wins (PINNED — Q-Token: INDEPENDENT coordinator generation + STA single-thread ordering; NOT shared with `NavigationController._generation`).** `RerenderAsync` claims `int myGen = ++_generation` before the await and re-checks `myGen == _generation` after the gateway resolves; a superseded re-render (a newer switch, OR a navigation that has since landed) is dropped (no `ShowMarkdown`). This mirrors `NavigationController.RunAsync` (lines 152–156) but with the coordinator's OWN counter — no `NavigationController` edit. **The nav×switch precedence rule:** `SetCurrentPage` (called on every navigation render) ALSO `++_generation`, so a navigation invalidates any in-flight re-render — nav supersedes a pending re-render, a newer switch supersedes an older switch, and the last action's correct page is always shown (see "RESOLVED — Q-Token" + the state table rows 9–11). Both the navigation render-sink and the switch re-render run on the single STA UI thread, so the two independent counters + STA serialization are sufficient and strictly simpler than a shared counter.
- **Nothing held yet → no-op.** A switch before any page loads has no held markdown → `RerenderAsync` returns without rendering (no fetch, no crash). First-run default is Basic, so even a pre-load switch is harmless.

The coordinator is factored so the no-re-fetch + last-wins logic is `[Fact]`-testable WITHOUT a window: inject a `PersonalizationGateway` (over a fake `ILlmClient`) + a recording `Action<string,Uri>` render-sink; assert the sink input + that no fetch delegate was invoked (the coordinator holds no fetcher at all — zero-re-fetch by construction; a counting fake fetcher in the test asserts 0 if the test wires one to the navigation path).

### Key-entry UX (BYO-key — the testable invariants)
