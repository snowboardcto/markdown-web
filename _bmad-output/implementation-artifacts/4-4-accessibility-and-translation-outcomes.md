### Q-decisions (PINNED — binding on the Step-4 TDD + Step-5 impl agents)

**Q-Lang-Plumbing (HOW the target language reaches the LLM) — RESOLVED: the ENGINE appends a language directive; the registry Translate prompt stays a stable constant.**
- **Decision.** `PersonaRegistry.Seed`'s Translate persona carries a stable, language-AGNOSTIC structure-preserving `SystemPrompt`. `PersonalityEngine.PersonalizeAsync` composes the EFFECTIVE system prompt it forwards to `ILlmClient.CompleteAsync`: for the Translate persona (identified by `persona.Id == "translate"`) with a non-blank `readerContext.PreferredLanguage`, it appends a deterministic directive (e.g. `persona.SystemPrompt + "\n\nTarget language: " + lang + ". Translate the page into " + lang + ", preserving all Markdown structure."`). For every other persona, OR Translate with a blank language, the engine forwards `persona.SystemPrompt` UNCHANGED (no behavior change for 4.1/4.2/4.3 — re-asserted by the inherited capturing-fake tests, which use non-Translate personas).
- **Justification.** (1) The capturing-fake routing proof (AC3) needs ONE deterministic place where the language is interpolated; the engine is that place — pure, no WPF, `[Fact]`-testable. (2) Keeping the registry prompt a constant means the 4-3 prompt-distinctness/no-placeholder tests are trivially extended for Translate without per-language registry variants. (3) `ILlmClient.CompleteAsync` ALREADY receives `ReaderContext` (it is in the signature) AND the `systemPrompt` — the directive rides the `systemPrompt` arg, which the capturing fake records, so the proof is on the EXACT seam 4-3 already asserts on.
- **Consequence (BINDING):** the only `PersonalityEngine` edit is the Translate-directive composition (a small, pure branch). `ILlmClient`/`ReaderContext`/`PersonalizationGateway`'s SIGNATURES are unchanged; the gateway change is sourcing `PreferredLanguage` (below) instead of hardcoding `null`.

**Q-Lang-Source (where the gateway gets PreferredLanguage) — RESOLVED: a `Func<string?>` injected into the gateway, sourced from the language-selection state. PINNED as an OPTIONAL ctor param defaulting to `() => null` so EVERY existing 4-1/4-2/4-3 call site compiles UNCHANGED (backward-safe).**
- **Decision.** `PersonalizationGateway` gains a SECOND, OPTIONAL constructor parameter `Func<string?>? preferredLanguage = null` AFTER the existing `Func<Persona> selectedPersona` (mirroring its shape). The body coalesces a null arg to the null-language source: `_preferredLanguage = preferredLanguage ?? (() => null);`. `ResolveMarkdownAsync` builds `new ReaderContext(PageUrl: pageUrl?.ToString(), PreferredLanguage: _preferredLanguage())` instead of the hardcoded `PreferredLanguage: null`. The App composes it with `() => _languageSelection.Current` (the language VM). For non-Translate personas the engine ignores the language (Q-Lang-Plumbing), so a non-null language is harmless to Basic/Cozy/Terminal/TL;DR/Plain.
- **EXACT ctor (PINNED — backward-safety contract):**
  ```csharp
  public PersonalizationGateway(
      PersonalityEngine engine,
      Func<Persona> selectedPersona,
      Func<string?>? preferredLanguage = null)   // NEW, OPTIONAL — defaults to null-language
  {
      _engine = engine ?? throw new ArgumentNullException(nameof(engine));
      _selectedPersona = selectedPersona ?? throw new ArgumentNullException(nameof(selectedPersona));
      _preferredLanguage = preferredLanguage ?? (() => null);   // total: a null arg => null language
  }
  ```
  and `ResolveMarkdownAsync` line 41 becomes `var context = new ReaderContext(PageUrl: pageUrl?.ToString(), PreferredLanguage: _preferredLanguage());`.
- **Backward-safety (VERIFIED against the real code — load-bearing).** The repo has **14 existing two-arg call sites** of `new PersonalizationGateway(engine, () => persona)` — `MainWindow.xaml.cs:57` plus `PersonalityRerenderSeamTests`, `PersonalizationSeamTests` (×2), `PersonalitySelectionViewModelTests` (×2), `VisibleChangeOnSwitchTests` (×3), `ApiKeyPromptViewModelTests` (×2), `StructuralRenderDiffersTests`, `StructuralDifferenceTests` (×3), `PersonalizationGatewayTests` (×3). Because the new param is OPTIONAL with a default, **all 14 keep compiling and behave EXACTLY as today** (language stays `null` → identical `ReaderContext` → identical engine behaviour → the 4-1/4-2/4-3 gateway/seam/distinctness tests stay byte-green). Only MainWindow is EDITED to pass `() => _languageSelection.Current` as the 3rd arg; the test call sites are left UNCHANGED except where a NEW 4.4 test deliberately exercises the language path. This is the safe migration: no signature break, no positional churn.
- **Justification.** Symmetric with the proven `Func<Persona>` seam; the gateway stays total and reads the language per resolve (so a language change is picked up by the very next re-render); no new `ReaderContext` field (it already has `PreferredLanguage`); the optional default eliminates the only credible 4-1/4-2 regression risk (a forced positional break across 14 call sites).

**Q-Audio-Trigger (how audio is invoked) — RESOLVED: an `"audio"` persona in the registry that the App routes to the SPEECH path (not a FlowDocument re-render).**
- **Decision.** `PersonaRegistry.Seed` gains an `"audio"` persona (DisplayName "Audio (Read aloud)"). Selecting it does NOT re-render the FlowDocument — the App's selection handler detects `picked.Id == "audio"` and routes to the audio path: extract the reading-order text from the HELD RAW markdown (`ReadingOrderExtractor.Extract`) and `ISpeechSynthesizer.SpeakAsync(text)`. The visible document is left as-is (the reader still sees the page they were on). To mark audio as a distinct OUTPUT KIND, the `"audio"` persona is `IsPassThrough = true` (so if it EVER flows through the gateway it is a harmless pass-through, never an LLM call) — but the App short-circuits it to the speech path BEFORE the gateway/coordinator.
- **WHAT THE SCREEN SHOWS when Audio is selected (PINNED — resolves the pass-through-vs-speak tension):** because `IsPassThrough = true`, if audio EVER reached the gateway it would just return the original markdown — but it MUST NOT render markdown AS the audio output; audio is a SPOKEN output kind, not a visual re-render. The App therefore SHORT-CIRCUITS audio BEFORE the gateway/coordinator: the **currently-rendered FlowDocument is left exactly as-is** (the held basic/last-persona render the reader was already viewing stays on screen, unchanged) AND speech plays over it. Selecting Audio **never blanks, clears, garbles, or replaces** the visible page — it neither re-renders nor pushes anything to the render-sink. (Pinned consequence: the audio route touches ONLY the `ISpeechSynthesizer`; it makes ZERO render-sink calls. A `[Fact]` asserts the audio route does not invoke the render-sink / coordinator.)
- **Justification.** (1) Reuses the EXISTING selector (no second toolbar control needed for audio) — audio is "a personality" per the manifesto/D2 framing. (2) `IsPassThrough = true` makes audio key-free and LLM-free by construction (D5: no key). (3) The App-side route keeps the speech I/O in `App` (D3) and the extractor pure in `Agent`.
- **Stop-before-speak (PINNED — resolves the open Stop()/overlap question; a total wiring requirement, NOT a new AC):** the audio route ALWAYS calls `ISpeechSynthesizer.Stop()` IMMEDIATELY BEFORE every `SpeakAsync(...)`, so a re-selection of Audio (or any new speak) never overlaps a still-playing utterance. The App ALSO calls `Stop()` on navigation away (a fresh page render) so speech does not bleed across pages. This is a deterministic, CI-assertable invariant: the fake `ISpeechSynthesizer` records a `StopCount`, and a `[Fact]` asserts the route invoked `Stop()` before `SpeakAsync` (StopCount ≥ 1 on a non-empty speak; and on a second consecutive read, Stop precedes the second Speak). Stop ordering relative to Speak is the binding contract. (A VISIBLE Stop/pause/Read-aloud TOGGLE control remains OUT of 4.4 — Deferred-Work-Log; only the programmatic Stop-before-speak wiring is in-scope.)
- **Note:** the audio persona is NOT in the four "structural" personas the 4-3 AC2/AC4 distinctness proofs cover, and (being pass-through with an empty/▸minimal prompt) it is excluded from the Translate/structural prompt-content assertions.
