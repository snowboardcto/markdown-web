# Story 5.2: Follow / Feed

Status: review

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a reader,
I want to **follow a vault** by subscribing to its **standard feed** in any feed reader — so that when the author adds a new `.md` page (and it goes live on push, FR-17), that new page **surfaces to me automatically in my feed** without an account, a login, or any per-reader backend,
so that I can keep up with authors I value the same way the open web has always done it: a subscribable feed URL.

## Context note — "Follow" on the open web is a STANDARD STATIC FEED, not a subscription backend

> **CRITICAL SCOPE FRAMING — read before writing any code.** The user chose "Run full sprint as-is" on a deliberately deferred post-MVP epic. The single epic-level AC for 5.2 (epics.md lines 443–453) is: *Given I follow a vault, When the author adds a new `.md` page, Then that page surfaces to me as part of the vault's feed (FR-16).* On the open web, **"following a vault" = subscribing to its feed URL in a standard feed reader**, and "a new page surfaces in the feed" = **the feed is regenerated at build and the new page appears in it after the author pushes (FR-17 publish-on-push makes it live).** This is the **commodity "follow" mechanism** — RSS/Atom — and per **NFR-7 (don't reinvent commodity plumbing)** and the project's **static-first, zero/low-JS** stack, that is exactly what this story ships. There is **NO accounts system, NO server-side subscription store, NO push-notification backend, NO database, NO new Azure Function.**
>
> **The genuinely-NEW surface for 5.2 is small and static:** (1) a build-time **feed document** (`/feed.xml`, RSS 2.0) generated from the vault, listing every page with its **canonical absolute URL** and a **date**, ordered newest-first; (2) **autodiscovery** — a `<link rel="alternate" type="application/rss+xml">` in every page's `<head>` (plus a small visible "Subscribe"/feed affordance in the chrome) so browsers and feed readers find the feed. Everything else is REUSED verbatim:
>
> - **Vault enumeration** — Story 2.5 already enumerates the whole vault: `getCollection('pages')` + the pure `buildIndexItems()` in `web/src/lib/index-entries.mjs` (empty-id filter, never-empty label, deterministic code-unit sort). The feed reuses the SAME `getCollection('pages')` source so "every page that is a page (FR-1)" is exactly the set the index already lists — they cannot drift.
> - **Canonical absolute URLs** — Story 5.1 already set `Astro.site = 'https://themarkdownweb.com'` (in `web/astro.config.mjs`) and builds each page's canonical absolute URL via `new URL(path, Astro.site)` in `web/src/layouts/Page.astro`. The feed reuses `Astro.site` + the SAME route path (`'/' + entry.id`) to emit each `<link>`/`<guid>` — the feed item URL and the page's `<link rel="canonical">` are the SAME value, by construction.
> - **The build/deploy pipeline** — Epic 1 (`deploy-web.yml`, Azure SWA) + Story 1.4 (live at `https://themarkdownweb.com`) already publish `dist/` on push. An Astro **static endpoint** route (`feed.xml.ts`) emits `dist/feed.xml` as a normal static asset on every build — so a pushed new `.md` rebuilds the feed and the new item is live. NO new infra.
>
> The scope of THIS story is intentionally lean: a pure feed-builder helper + a static endpoint route that serves `/feed.xml`, the autodiscovery `<link>` + a visible subscribe affordance in the shared chrome, a deterministic **date** per page, and tests proving the feed is valid RSS 2.0, lists every vault page with absolute canonical URLs + dates, a new page appears after rebuild, and the 157+ existing web specs do not regress.

## Acceptance Criteria

> **The single epic-level AC (epics.md Story 5.2, lines 449–453):** *Given I follow a vault, When the author adds a new `.md` page, Then that page surfaces to me as part of the vault's feed (FR-16).* Because "follow / feed" on the open web is the standard subscribable-feed mechanism, it is DECOMPOSED below into concrete, testable sub-ACs. AC1–AC3 are the genuinely-NEW static feed surface (feed generated, new-page-surfaces, valid spec). AC4 is autodiscovery + the visible subscribe affordance. AC5–AC7 are the standing scope/regression/CI guardrails. The "follow" act itself is the reader subscribing to the feed URL in their own feed reader — no product code, by design (AC5).

1. **[`/feed.xml` is generated at build, listing EVERY vault page with absolute canonical URLs + dates, newest-first (NEW — the feed itself)]** **Given** the vault (the current `content/**/*.md` set, the SAME set `getCollection('pages')` / Story 2.5's index enumerate), **When** the site builds, **Then** a static feed document is emitted to `dist/feed.xml` (served at `https://themarkdownweb.com/feed.xml`) that contains **exactly one feed item per vault page** — the same one-to-one completeness invariant as Story 2.5's index (count of feed `<item>`s EQUALS the number of `getCollection('pages')` entries after the empty-`id` filter; no page omitted, no extra item, no self-referential item for the index route `/`). For **each** item:
   - the item **link AND guid are the page's canonical absolute URL** — `new URL('/' + entry.id, Astro.site).href` (e.g. `/gear-guide` → `https://themarkdownweb.com/gear-guide`, `/sub/page` → `https://themarkdownweb.com/sub/page`) — **byte-equal** to the value `Page.astro` declares in that page's `<link rel="canonical">` (REUSE `Astro.site`; do NOT re-implement slug/URL logic — derive from `'/' + entry.id` exactly as `buildIndexItems` does). The `<guid>` carries **`isPermaLink="true"`** and equals the `<link>`: this is the **build-stable item identity** a feed reader keys "already seen" off of (it derives from `entry.id`, so it does NOT change between builds — unlike `<pubDate>`, which moves to the build date for undated pages; see the AC2 date-policy second-order note). Do NOT use a build-derived or random guid;
   - the item **title** is the page's human-readable title via the SAME precedence the index uses (`entry.data.title || slugToTitle(entry.id) || entry.id` — REUSE `buildIndexItems`/`slugToTitle`, do NOT re-derive);
   - the item carries a **date** (`<pubDate>`) per the deterministic date policy in AC2's Dev-Notes decision.
   - The feed also carries channel-level metadata: `<title>` (the site/vault name), `<link>` (the site root, `Astro.site.href`), `<description>`, `<language>`, a `<lastBuildDate>`, and the **`<atom:link rel="self">`** pointing at the feed's own absolute URL (`https://themarkdownweb.com/feed.xml`) — required for a well-formed, autodiscoverable RSS 2.0 channel.
   - **Ordering:** items are ordered **newest-first by date** (most-recent `<pubDate>` first), with a **deterministic, CI-stable tie-breaker** (e.g. code-unit by `entry.id`, mirroring 2.5's ICU-independent sort) so the feed order is reproducible build-to-build and dev↔CI (NOT `getCollection` iteration order, NOT a locale-dependent compare).
   - **Degenerate cases (mirror 2.5):** an **empty vault** (zero `.md` after the empty-`id` filter) yields a **valid, well-formed RSS channel with zero `<item>`s** (NOT a crash, NOT malformed XML) — the channel-level `<title>/<link>/<description>/<lastBuildDate>/<atom:link rel="self">` are all still present and well-formed even with no items; a **single-page vault** yields exactly one `<item>`.
   - **All-equal-date boundary (Boundary Sweep — the ordering tie-break must actually fire):** when **every** page resolves to the SAME date (the common case until any page carries a frontmatter `date` — see AC2: all pages fall back to the one build/channel date), newest-first by date is a total no-op, so the **code-unit `entry.id` tie-break is the ONLY thing that makes the order deterministic**. The order in that case MUST equal `buildIndexItems`' code-unit `id` order exactly (so feed order == index order when undated), and MUST be identical dev↔CI. This is the realistic default state of the current vault, not a corner case — test it explicitly.
   *(AC1 — feed.xml generated at build, one item per vault page, absolute canonical URLs + dates, deterministic newest-first order; epics.md Story 5.2 "the vault's feed"; FR-1 file-as-page, FR-16; REUSE getCollection('pages') + Astro.site + buildIndexItems/slugToTitle.)*

2. **[A new `.md` page surfaces in the feed after rebuild (NEW — the core epic outcome, made testable)]** **Given** the feed is generated from `getCollection('pages')` at build time (AC1), **When** the author **adds a new `.md` page** to the vault and the site **rebuilds** (which on push is automatic via FR-17 publish-on-push / `deploy-web.yml`), **Then** that new page **appears as a new `<item>` in `/feed.xml`** — with its canonical absolute URL, title, and date — so a reader subscribed to the feed URL sees the new page surface in their feed reader (the FR-16 outcome). The feed is **generated, never hand-maintained** (it reads the collection, exactly like the 2.5 index), so "add a file → it shows up in the feed" holds with no manual edit — the same "add a file, get a page (and it shows up in the index)" promise from Story 2.5, extended to the feed.
   - **DECIDE-AND-DOCUMENT the date policy (record in Dev Agent Record) — this is load-bearing for "new page surfaces" and for newest-first ordering:** the vault frontmatter has **NO schema today** (`web/src/content.config.ts` defines no `schema`, so a `date` field is NOT guaranteed or validated). Define a **deterministic, total date-resolution** per page with a documented fallback chain, e.g.: (i) a frontmatter `date`/`pubDate` if present and parseable as a valid `Date`; else (ii) a stable fallback that is reproducible in CI (e.g. a fixed channel build date / the build timestamp, applied uniformly) — **pick one and document the exact precedence + the CI-determinism rationale**. **Do NOT** rely on filesystem mtime if it is not reproducible across a fresh CI checkout (a clean clone resets mtimes), unless you explicitly account for it; prefer frontmatter-or-build-date for CI stability. Whatever you choose, a page with no date MUST still appear in the feed (never dropped) and the order MUST be CI-stable.
   - **The test must genuinely exercise this** (not assert a tautology): add a throwaway `.md` fixture (or assert against a fixture already present), rebuild, and assert the feed now contains an `<item>` whose `<link>` is that page's canonical absolute URL — then remove the throwaway (or keep a committed fixture and pin it). Mirror Story 2.5 Task 5's "add a `.md`, rebuild, confirm it appears, then remove it" verification.
   *(AC2 — new page surfaces in the feed after rebuild; epics.md Story 5.2 "When the author adds a new `.md` page, Then that page surfaces to me as part of the vault's feed"; FR-16 + FR-17; generated-from-collection, not hand-maintained.)*

3. **[The feed is valid, well-formed RSS 2.0 served with the correct content type (NEW — spec correctness)]** **Given** `/feed.xml` is the commodity follow mechanism a standard feed reader consumes, **When** it is fetched, **Then** it is **well-formed XML** and **valid RSS 2.0**: a single `<rss version="2.0">` root with the `xmlns:atom` namespace declared, exactly one `<channel>`, the required channel children (`<title>`, `<link>`, `<description>`), each `<item>` with the RSS-required/expected children (`<title>`, `<link>`, `<guid>`, `<pubDate>` in RFC-822 date format), and the served response has **`Content-Type: application/rss+xml`** (or `application/xml`/`text/xml` with a documented rationale). All dynamic text (titles, descriptions) is **XML-escaped** (`&`, `<`, `>`, `"`, `'`) so a page title containing `&` or `<` cannot produce malformed XML or an injection — escape in the pure builder, test with a fixture title containing `&`/`<`.
   - The feed is **pure static build-time output** — no `client:*`, no runtime JS, no server-side per-request generation (it is an Astro static endpoint emitted into `dist/`). It must survive being served as a plain static asset by Azure SWA.
   - **Edge-case floor (Boundary & Edge Case Sweep):** (i) a title with XML-special chars (`Tom & Jerry <draft>`) → escaped, still well-formed; (ii) a page with no frontmatter title (H1-only / slug-only) → a non-empty `<title>` via the `slugToTitle || entry.id` fallback (never an empty `<title></title>`); (iii) the index route (`entry.id === ''`) is filtered (no item with link `https://themarkdownweb.com/` duplicating the channel `<link>`); (iv) a unicode/percent-encoded slug → a well-formed absolute URL in `<link>`/`<guid>` (reuse the same `'/' + entry.id` the index uses; no double-encode); (v) RFC-822 `<pubDate>` formatting is locale-independent (use a fixed-locale/UTC formatter, NOT `toLocaleString`, so dev↔CI match); (vi) an **ampersand already inside a URL** (a slug or query-like segment producing `&` in `<link>`/`<guid>`) is escaped to `&amp;` THE SAME as title text — escape happens once, at XML-serialization of every dynamic node, so URLs and titles share one escaper and a `&` is never emitted raw nor double-escaped (`&amp;amp;`); (vii) a **non-string / non-Date frontmatter `date`** (a number, an already-parsed `Date`, an unparseable string like `"someday"`, `null`) MUST NOT throw and MUST fall through to the documented fallback (total `resolveDate`, never `Invalid Date` in a `<pubDate>`).
   *(AC3 — valid, well-formed RSS 2.0, correct content type, XML-escaped, static; NFR-7 commodity plumbing; RSS 2.0 spec.)*

4. **[Autodiscovery + a visible subscribe affordance so readers/feed-readers find the feed (NEW — discoverability)]** **Given** "following" requires the reader to find the feed URL, **When** any page is served, **Then** the document `<head>` declares **feed autodiscovery**: `<link rel="alternate" type="application/rss+xml" title="..." href="<absolute /feed.xml URL>">` (the standard mechanism browsers + feed readers use to detect a page's feed) — added ONCE to the shared layout `web/src/layouts/Page.astro` so every page (`[...slug]`, the index, the 404) inherits it, exactly like the 5.1 canonical `<link>` lives there. The `href` is the feed's **absolute** URL built from `Astro.site` (`new URL('/feed.xml', Astro.site).href`) so it is unambiguous, and is **static HTML** (no `client:*`, no `<script>`) so it survives JS-disabled and is crawlable (FR-7/NFR-4).
   - **Plus a visible affordance** in the existing shared chrome: a labeled, keyboard-reachable **"Subscribe"/"Feed"** link in `web/src/components/SiteHeader.astro` (the sticky banner that already holds the wordmark · "the vision" · "Copy link" · "Get the client"), pointing at `/feed.xml` (or the absolute URL). A real `<a>` (it navigates to a static asset — no JS needed, unlike 5.1's copy button), accessible-named (e.g. "Subscribe" / "RSS feed"), AA-contrast, mirroring the existing ghost-link/CTA a11y pattern (NFR-6/UX-DR9). Decide-and-document the label + placement (header vs pitch-card).
   - This is **analogous to 5.1's canonical work**: a single shared-chrome edit reaching every page; the autodiscovery `<link>` is the machine-facing discovery path, the visible link is the human-facing one.
   *(AC4 — `<link rel="alternate" rss>` autodiscovery in `<head>` + a visible subscribe affordance in the chrome; epics.md "follow a Vault"; UX-DR2/UX-DR9; the standard feed-discovery mechanism.)*

### The load-bearing decisions

- **Decision — feed format = RSS 2.0, hand-emitted (NO library).** RSS 2.0 is the commodity follow format every feed reader consumes. NFR-7 + 2.5's "no new dependency for the index" precedent → emit the XML from a pure `feed.mjs` builder rather than adding `@astrojs/rss`. (Atom is acceptable too; RSS 2.0 is the recommended default. Pick one, document it.) [Source: NFR-7; 2.5 no-dep precedent; AC3/AC5]
- **Decision — `/feed.xml` via an Astro static endpoint, NOT `api/`.** The feed is build-time static output in `dist/`, served by Azure SWA as a plain asset — it is NOT content negotiation, so it does NOT belong in `api/` (architecture: "api/ only negotiates"). [Source: architecture "api/ only negotiates"; AC5]
- **Decision (DOCUMENT) — date policy.** Frontmatter has no schema, so define a total, CI-stable date resolution (frontmatter `date`/`pubDate` if valid → else build/channel date), RFC-822 UTC formatting, newest-first with code-unit `id` tie-break. A page with no date is never dropped. Record the precedence + CI-determinism rationale. [Source: AC2; web/src/content.config.ts no-schema]
  - **Second-order effect of the build-date fallback (anticipate, don't be surprised in review):** because today **no page carries a frontmatter date** (no schema), the fallback applies to ALL of them, so on every build EVERY undated item's `<pubDate>` equals the single channel build date. Two downstream consequences, both acceptable and intended — record them so a reviewer doesn't read them as bugs: (1) **Ordering degenerates to the `id` tie-break** (the all-equal-date boundary in AC1) — feed order == index order until an author adds a real `date`; this is correct, not a defect. (2) **`<pubDate>` is NOT a stable per-item identity across builds** (it moves to each new build date), so item identity for feed readers MUST come from **`<guid isPermaLink="true">` = the canonical absolute URL** (stable across builds because it derives from `entry.id`), NOT from `<pubDate>`. A feed reader keys "have I seen this item?" off the guid; the canonical-URL guid keeps already-seen pages from re-surfacing as "new" every rebuild even though their `pubDate` changed. This is the load-bearing reason the guid is the canonical URL, not a build-derived token. (3) **Choose a fixed channel build date over `Date.now()` per build** if you want byte-stable feed output across no-content rebuilds (a `Date.now()` fallback re-writes every `<pubDate>` + `<lastBuildDate>` on every CI run, so the `dist/feed.xml` diff is never empty even when content is unchanged — pick one and state the trade-off: build-timestamp = honest "regenerated now" semantics but noisy diff; fixed/derived date = quiet diff but less literal). Either is CI-deterministic *within a single build*; document which and why.
- **Decision (DOCUMENT) — subscribe affordance label + placement.** Header (`SiteHeader.astro`) for reach, label "Subscribe"/"RSS feed". [Source: AC4]

### Pre-mortem — how 5.2 ships broken (and the guard that prevents it)

> _Assume it is post-merge and 5.2 is judged a failure. Failure modes ranked, each mapped to an in-story guard (no new scope)._

1. **Set drift — the feed lists a different set than the index.** The feed re-implements its own enumeration/filter instead of reusing `buildIndexItems`, so it omits or duplicates a page (e.g. lists the index `/`). → **Guard:** AC1 mandates REUSE of `getCollection('pages')` + `buildIndexItems`; Task 4 asserts the feed item count EQUALS the page count one-to-one and that no item links the bare site root.
2. **URL drift — feed item URL ≠ the page's canonical.** The builder constructs its own URL (picking up a trailing slash / wrong host) instead of `new URL('/' + entry.id, Astro.site)`. → **Guard:** AC1 mandates the same `Astro.site` + `'/' + entry.id`; Task 4 byte-compares a feed item link to the page's `<link rel="canonical">`.
3. **New page does NOT surface — the date policy drops undated pages or the order is non-deterministic.** A page with no frontmatter date is filtered out, or ordering depends on `getCollection` iteration / host locale. → **Guard:** AC2 mandates a total date policy (undated pages still appear) + a code-unit tie-break; Task 4 add/rebuild/confirm; Task 1 UTC formatter + documented precedence.
4. **Malformed XML — a title with `&`/`<` breaks the feed.** Unescaped dynamic text yields invalid XML a feed reader rejects. → **Guard:** AC3 mandates escaping; Task 1 escapes in the builder; Task 4 pins an `&`/`<` fixture and parses the feed.
5. **Scope creep into a backend.** A "followers" store, an email/notification stub, an `api/` endpoint, or `@astrojs/rss` sneaks in. → **Guard:** AC5 forbids backend/store/notifications/Function/dependency; Task 5 asserts file scope + no new dep; the scope-boundary section enumerates the NOT-list.
6. **Discoverability gap — readers can't find the feed.** No autodiscovery `<link>`, or it's JS-dependent. → **Guard:** AC4 puts a static `<link rel="alternate" rss>` in `<head>` + a visible subscribe link; Task 4 asserts both, JS-disabled.
7. **Web regression — the 192 specs go red.** Adding the subscribe link to `SiteHeader.astro` shifts a chrome assertion broadly. → **Guard:** AC6 — additive; reconcile ONE chrome spec minimally if needed; verify the `dist/` diff is only the feed + the two additive head/chrome lines.
8. **"Surfaces" inverts into "everything re-surfaces" — every page re-pings as new on every rebuild.** Because all pages are undated, if item identity is keyed off `<pubDate>` (or a build-derived guid), every rebuild re-stamps every item with a new date/guid, so a reader's feed reader marks the WHOLE vault unread on every push — the opposite of FR-16's "the NEW page surfaces." This is the Second-Order trap of the build-date fallback. → **Guard:** AC1 mandates `<guid isPermaLink="true">` = the canonical absolute URL (derived from `entry.id`, build-stable); AC2's date-policy note makes the guid (not `pubDate`) the identity; Task 1 sets the canonical-URL guid and Task 4 asserts `<guid> == <link> ==` the page's `<link rel="canonical">` and that the guid is byte-stable across a rebuild.

**Date Policy Decision (AC2):**
- Precedence: `entry.data.date` (string or Date, parseable) → `entry.data.pubDate` (string or Date, parseable) → `channelBuildDate` (single `new Date()` per `buildFeed` call).
- Numbers, unparseable strings, null, undefined all fall through to the fallback — total, never throws, never emits "Invalid Date".
- Using a single `channelBuildDate` per call (not `Date.now()` per item) keeps the feed diff quiet when content is unchanged: all undated items within one build share the same date.
- All dates formatted as RFC-822 UTC via a fixed-locale hand-formatter (no `toLocaleString`) — dev↔CI identical.
- Second-order consequence documented: because today no page has a frontmatter date, ALL items share the build date, so ordering degenerates to code-unit `entry.id` tie-break (feed order == index order). This is intentional and expected.
- **GUID = canonical URL** (not pubDate or build-derived token): since pubDate changes each build for undated pages, the guid must be the build-stable canonical URL so feed readers don't re-surface all items as new on every rebuild (pre-mortem #8).
