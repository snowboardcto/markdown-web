## Why these tests fail now (the RED baseline)

Today, with no rehype media plugin and no vault-media copy step:

1. Relative media `src`/`poster` are emitted **literally** (`media/powder.jpg`,
   `../media/powder.jpg`, `diagram.png`) — NOT rewritten to a served
   root-absolute `/media/powder.jpg`.
2. `content/media/**` is never copied into `dist`, so `GET /media/powder.jpg`
   returns **404** (asset not served, no image content-type, no inline render).

## Fixture authoring decision (important)

The 2.4 media fixtures in `content/x.md` and `content/sub/page.md` are authored
as **raw HTML** `<img>` / `<video>` / `<audio>` / `<source>` rather than markdown
`![]()`. Two reasons:

1. **Same code path.** The implementation is a rehype (HAST) transformer that
   visits media ELEMENT nodes by `tagName` (`img`/`video`/`audio`/`source`),
   independent of whether the element came from `![]()` or raw HTML. Raw HTML
   exercises the exact same rewrite branch the plugin will run.
2. **No wrong-reason build crash.** Astro's default `astro:assets` pipeline
   **hard-fails the whole build** (`ImageNotFound`) on a missing markdown `![]()`
   image, and **auto-rewrites** existing markdown images to optimised
   `/_astro/*.webp` hashed paths. Either would take the build (and the existing
   66 specs) down for the wrong reason in the RED phase. Raw HTML media tags are
   NOT processed by `astro:assets`, so the AC6 missing-asset reference
   (`media/missing.jpg`) builds cleanly and emits a literal `src` — the true
   feature-absent baseline. The implementation step disables/bypasses
   `astro:assets` for content media so markdown `![]()` also flows through the
   plugin; these specs assert the served output and pass either way.

Verified empirically during authoring: the built `dist/x/index.html` contains
exactly one `<h1>` and one `<table>`, zero `/_astro` references, and the full
fixture body (so `ac1`/`ac6`/`2-2-theme` assertions on `/x` stay valid).

## The real binary fixture assets

- `content/media/powder.jpg` — a **valid 331-byte baseline JPEG** (JFIF 1.01,
  8-bit, grayscale, standard Annex-K Huffman tables, SOI `FF D8` … EOI `FF D9`).
  `file(1)` reports: `JPEG image data, JFIF standard 1.01, baseline, precision 8`.
  Browser-decodable, so the `naturalWidth > 0` / 200 / `image/*` assertions are
  real, not stubbed.
- `content/sub/diagram.png` — a **valid 69-byte 1×1 truecolor PNG** (the AC2
  sibling-resolution + nested-path-copy fixture). `file(1)`: `PNG image data,
  1 x 1, 8-bit/color RGB`.

### How the JPEG was generated (no Pillow available in the env)

A pure-`python3` script (no third-party deps) writes the JPEG byte-by-byte to
the JFIF/JPEG spec: `SOI` → `APP0`(JFIF) → `DQT`(all-ones luminance quant table)
→ `SOF0`(baseline, 1×1, 1 grayscale component) → two `DHT`s (the standard Annex-K
luminance DC + AC Huffman tables) → `SOS` → a 1-MCU entropy-coded scan (DC diff 0
+ AC EOB, bit-packed with `0xFF` byte-stuffing) → `EOI`. Using the *standard*
Huffman tables (not custom ones) is what makes it decodable by browsers. The PNG
was likewise generated by a dependency-free `python3` script
(`sig` + `IHDR` + zlib-deflated `IDAT` + `IEND`, with correct CRC32s).

> The generator scripts were run inline via the Bash tool; they are not committed
> (only the resulting binary assets are). To regenerate, re-run the byte-builder
> shown in the session transcript.
