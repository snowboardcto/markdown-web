# Story 2.7: Content negotiation — one URL, two representations

Status: review

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As an agent or native client,
I want the same page URL to serve raw markdown when I ask for it (`Accept: text/markdown`) while browsers (`Accept: text/html`) still get the rendered HTML page,
so that the native client (Epic 3) and crawlers/agents can fetch the canonical `.md` source from the exact same URL a human reads, with caches keeping the two representations apart.

## Context note (this is the FIRST `api/` story — not a `web/` story)

Stories 2.1–2.6 built the `web/` Astro static site. Story 2.7 is the **content-negotiation backend** (FR-14) and is the **first work in `api/`** (Azure Functions). It does NOT change the Astro rendering pipeline, the content collection, the slug logic, or any existing `web/` behaviour — the 157 existing `web/tests/*.spec.ts` MUST stay green untouched (AC8). The deliverable is: an Azure Function that, for a page URL, returns the raw `.md` on `Accept: text/markdown` and lets the browser HTML path serve on `Accept: text/html`, with `Vary: Accept` so caches never serve markdown to a browser.

## Acceptance Criteria

> **AC1/AC7 — DECISION (Option 2, pragmatic; naethyn, 2026-06-21): true same-URL `Accept` negotiation is DEFERRED.** Azure Static Web Apps route rules cannot branch on the request `Accept` header (a documented platform limitation — `rewrite`/`redirect` are path-only), so routing a page-URL `text/markdown` request to the Function is not expressible in `staticwebapp.config.json`. Per the decision we DO NOT front page URLs with the Function. Instead: the page URL `/<slug>` stays **pure static HTML** (the fast static host, live site untouched), and **raw markdown is exposed NOW at the Function endpoint `/api/negotiate/<slug>`** with `Content-Type: text/markdown; charset=utf-8` + `Vary: Accept` (and a sensible HTML/redirect response for other Accepts at that endpoint). The native client (Epic 3, Story 3.2) — the only consumer — fetches markdown from `/api/negotiate/<slug>`. The original AC1/AC7 below describe the IDEAL same-URL end-state; it is recorded as KNOWN-DEFERRED in `deferred-work.md` and revisited when Epic 3 lands.

1. **Given** a `.md` page URL served by the site (a page that exists in the Astro build, e.g. `/x`, `/gear-guide`, `/sub/page`, `/my-notes`), **When** a request arrives with `Accept: text/markdown` **[Option 2: at the Function endpoint `/api/negotiate/<slug>`; the page URL `/<slug>` stays static HTML]**, **Then** the response is **HTTP 200** with the **raw `.md` source bytes** of that page's `content/**/*.md` file, `Content-Type: text/markdown; charset=utf-8`, and `Vary: Accept`. The raw markdown is the verbatim file content (the same bytes a build-time read of `content/x.md` would yield) — NOT re-rendered, NOT the HTML, NOT a re-serialized AST. *(AC1 — `Accept: text/markdown` → raw `.md` + `Content-Type: text/markdown; charset=utf-8` + `Vary: Accept`; epics.md Story 2.7 line 274–276, FR-14)*

2. **Given** the same page URL, **When** a request arrives with `Accept: text/html` (a normal browser) — OR with no `Accept`, OR an `Accept` that does not **positively** include `text/markdown` (e.g. `*/*`, `text/*`, `text/html,application/xhtml+xml,...,*/*`) — **Then** the request is served the **HTML page** (the static Astro page), NOT raw markdown. HTML is the DEFAULT representation: markdown is returned ONLY when the client explicitly opts in via a `text/markdown` media-range that the markdown branch actually wins. **Accept-parsing correctness bar (RFC 9110 §12.5.1):** the parser MUST (a) split on `,`, strip optional whitespace, and parse each media-range as `type/subtype` + `;`-delimited params; (b) match media-types **case-insensitively** (`Text/Markdown` ≡ `text/markdown`); (c) **ignore non-`q` params** so `text/markdown;charset=utf-8` and `text/markdown; charset=UTF-8` still match markdown; (d) honor **q-values** — a media-range with `q=0` is an explicit REJECTION (e.g. `text/markdown;q=0, text/html` → HTML; `*/*, text/markdown;q=0` → HTML), and when both `text/markdown` and an HTML/`*` range are acceptable, markdown wins ONLY if its effective q-value is strictly greater than 0 AND ≥ the best HTML/`*` q-value (ties between an explicit `text/markdown` and a wildcard `*/*` resolve to **markdown**, because `text/markdown` is the more specific range; a tie against an explicit `text/html` of equal q resolves to **HTML** — markdown is opt-in, so an equal-preference browser stays on HTML); (e) treat `text/*` and `*/*` as **NON-opt-in** (a browser sending `*/*` or `text/*` MUST get HTML — wildcards never trigger the markdown branch on their own); (f) on a **malformed / unparseable `Accept`** (empty media-range, bare `;`, non-ASCII garbage, missing subtype) → fail **safe to HTML**, never 400/500 and never markdown. *(AC2 — Accept-header parsing rigor: case-insensitive, param-tolerant, q-value-aware, `q=0`=reject, wildcards never opt-in, malformed → safe HTML default; epics.md Story 2.7 line 277 "a browser receives the HTML page", FR-14, RFC 9110 §12.5.1)*

3. **Given** the two representations share one URL, **When** ANY negotiated response is emitted (both the markdown branch AND the HTML branch on the negotiated route), **Then** it carries `Vary: Accept` so that a shared/CDN cache keys the cached entry on the `Accept` header and **never serves a stored markdown body to a browser** (or a stored HTML body to a markdown client). The `Vary: Accept` header is present on BOTH branches of the negotiated route — emitting it only on the markdown branch is insufficient, because a cache that stored the HTML branch without `Vary` could replay it for a markdown request (and vice-versa). **Edge/CDN-correctness sub-bar:** (a) the `Vary` value MUST be exactly `Accept` (case-insensitive token, `Vary: Accept`), and if any other vary dimension is ever added it MUST be comma-joined into the SAME header, not collapsed to `Vary: *` (which would make every response uncacheable) and not duplicated; (b) `Vary: Accept` MUST be advertised by **whatever surface actually serves each branch** — including the SWA **static host** when it serves the HTML branch (set via `staticwebapp.config.json` `headers`/`responseOverrides`), because an un-keyed HTML asset cached at the Azure CDN/Front-Door edge would be replayed for a `text/markdown` request regardless of what the Function returns (and vice-versa); (c) the two representations are byte-distinct bodies at one URL, so the **caching policy MUST NOT be more permissive than `Vary` can protect** — if a `Cache-Control` is set, it is paired with `Vary: Accept` on both branches so no shared cache can satisfy a markdown request from a stored HTML entry; document the chosen `Cache-Control` (e.g. inheriting SWA defaults vs an explicit value) and confirm it is `Accept`-keyed. The DoD check: a CDN that saw a browser (`Accept: text/html`) request first MUST still forward/serve the markdown branch for a subsequent `Accept: text/markdown` request to the same URL, and never the reverse. *(AC3 — `Vary: Accept` on BOTH branches AND on the static-host HTML surface; `Vary` value-correctness (`Accept`, never `*`); `Cache-Control` no more permissive than `Vary` protects; caches/CDN never cross-serve representations; epics.md Story 2.7 line 278 "caches never serve a markdown response to a browser", architecture Cross-Cutting Concern #5 "content-negotiation caching — `Vary: Accept` correctness")*

4. **Given** the function maps a request URL → its raw `.md` source, **When** it resolves the source file, **Then** it uses the **SAME slug derivation the web build uses** so the markdown URL and the HTML URL are identical: a page at route `/<slug>` returns the `content/**/*.md` whose `pathToSlug` (`web/src/lib/slug.mjs`: drop `.md`, github-slug each segment → lowercase, `/index` collapse) equals `<slug>`. The mapping is exact and case/space-normalized identically to the HTML route (`content/My Notes.md` → `/my-notes`, `content/sub/page.md` → `/sub/page`, `content/sub/index.md` → `/sub`). It must NOT re-implement a *different* slug rule that drifts from `[...slug].astro` / `slug.mjs`. *(AC4 — URL→`.md` mapping uses the SAME slug logic as the web route (no drift); architecture "single source of truth `content/`; both web build & api consume the same `.md`")*

5. **Given** a request for a slug that has **no** corresponding `content/**/*.md` page (e.g. `/does-not-exist`) on the markdown branch, **When** `Accept: text/markdown` is sent, **Then** the function returns **HTTP 404** (not a 200 with an empty/garbage body, not a 500), with a clear, non-crashing not-found response; it never serves another page's markdown, never path-traverses outside the vault, and never leaks filesystem paths. **Slug→file security bar (defense-in-depth, ordered):** (a) resolve the slug to a `.md` file ONLY through the **closed slug→file map** built by the shared `pathToSlug` logic (Task 2) — i.e. a request slug is a **lookup key**, never concatenated into a filesystem path — so the file set is exactly the build's known pages and an unknown key is a clean 404 by construction; (b) BEFORE/at lookup, reject hostile keys: URL-decode the slug **exactly once** then reject any of — `..` or `.` path segments, embedded NUL (`%00`), a leading separator / drive letter / UNC / absolute path (`/etc/passwd`, `\\`, `C:\`), a backslash, an encoded separator (`%2F`, `%2f`, `%5C`), double-encoding (`%252F`), or any residual `/`-bearing segment that does not correspond to a real nested-slug key; a decoded key still containing traversal tokens → 404, never a second decode pass; (c) **after** any path is formed for the chosen file, canonicalize (`path.resolve`/`realpath`) and assert the result is **lexically contained within the resolved vault root** (`resolved.startsWith(vaultRoot + sep)`) — if it escapes, 404, regardless of how it got there (belt-and-braces even though (a) makes concatenation unnecessary); (d) honor the build's **fail-loud collision contract** — the map is collision-free by construction (the build throws on two files → one slug, per `[...slug].astro` lines 48–60), so the map MUST never silently pick one of a colliding pair; if a manifest is used it carries the same guarantee; (e) **no path leak / no oracle** — the 404 body and headers are identical for "valid slug, no page" and "hostile slug" (same status, same body, no echoed slug, no filesystem path, no stack trace, no distinguishing timing/length signal that lets an attacker probe the vault layout). Encoded-separator / traversal smuggles in the slug (`%2F`, `..`, `%2Fetc%2Fpasswd`, `..%2f..%2fetc%2fpasswd`, `%252e%252e`) all resolve to an indistinguishable 404, never to a file outside the vault and never to a different page's markdown. *(AC5 — missing slug → 404; closed-map lookup (key, not path); decode-once + reject `..`/`%2F`/`%00`/absolute/UNC/double-encoded; post-resolve vault-containment assert; collision fail-loud preserved; uniform 404 with no path leak / no probe oracle; epics.md broken-link discipline (2.3 F1 / 2.4 smuggle-guard precedent), FR-14 safety)*

6. **Given** the raw `.md` must round-trip faithfully for the native client's Markdig parser (Epic 3), **When** the markdown body is returned, **Then** it is **byte-faithful** to the source file: UTF-8, no BOM injection, no CRLF↔LF rewriting beyond what the source contains, no trailing-newline mutation, no markdown re-formatting. A client fetching `/x` with `Accept: text/markdown` receives bytes equal to `content/x.md` on disk. **Byte-fidelity implementation bar:** (a) the file is read and served as a **`Buffer`/raw bytes**, NOT read-as-string-then-re-encoded — a `readFile(path)` (no `'utf8'` flag) → body passthrough, so the runtime never normalizes line endings, never re-encodes, and never re-serializes; (b) **no BOM stripping AND no BOM injection** — if the source has a leading `EF BB BF` it is preserved verbatim, and the function never prepends one (note: bundling/copy steps and editors can silently add a BOM — the Task-1 copy step MUST be byte-exact, e.g. `cp`/`fs.copyFile`, not a read-as-text rewrite, mirroring `copy-vault-media.mjs`'s verbatim discipline); (c) the response is sent in a mode that does NOT mutate the body — set `Content-Type: text/markdown; charset=utf-8` but ensure the Functions runtime treats the body as binary/`Buffer` (avoid any auto-`JSON.stringify` or string-charset transform on the body path); (d) **`Content-Length`, if emitted, equals the source byte length** (`Buffer.byteLength`), and there is no `Transfer-Encoding`/compression step that re-frames the bytes such that a `Content-Length` mismatch or altered bytes could occur — if the platform gzips, the decoded body still byte-equals the source; (e) the handler unit test asserts **`Buffer.equals(served, fs.readFileSync('content/x.md'))`** (raw-buffer equality, AC8a) — a string-compare can hide BOM/encoding drift, so the assertion is buffer-level. *(AC6 — raw `.md` byte fidelity: read-as-Buffer/no re-encode, preserve+never-inject BOM, no CRLF/trailing mutation, `Content-Length`==source bytes, buffer-equality test; architecture FC-1/GFM canonical-source single-source-of-truth, Epic 3 Markdig consumer)*

7. **[Option 2 — RESOLVED-AS-DEFERRED for same-URL; see the AC1/AC7 banner above.]** The deterministic mechanism is: raw markdown is served at the linked Function endpoint `/api/negotiate/<slug>` (reachable via the SWA managed-API route), while the page URL `/<slug>` is served as static HTML by the static host. The `staticwebapp.config.json` carries ONLY correct, working config (the `/api/negotiate/*` route advertises `Vary: Accept`; no page-URL→Function rewrite that would falsely imply same-URL negotiation), and `api_location: 'api'` is wired into `deploy-web.yml`. The chosen routing approach (e.g. SWA route rule forwarding page routes to the Function which then negotiates on `Accept`, OR the Function fronting all page routes and proxying/redirecting to static HTML for the browser branch) is recorded with its trade-offs. No secret/subscription-id is hard-coded (consistent with Story 1.2 IaC discipline). *(AC7 — SWA routing wires `Accept`-based negotiation: committed `staticwebapp.config.json` + `api_location` in deploy-web.yml; mechanism + trade-offs documented; architecture "Azure SWA + linked Functions API + `staticwebapp.config.json`", FR-17/18 deploy spine)*

### Architecture compliance / guardrails

- **`api/` only negotiates** — browsers → static HTML (Astro/SWA); clients → raw `.md`. It holds NO content of its own; it READS the vault `.md` (bundled or emitted per Task 1) and returns it verbatim. Do NOT render, transform, or re-serialize the markdown in the API. [Source: architecture "api/ only negotiates"; api/README.md]
- **`content/` is the single source of truth** — both `web/` (build time) and the native client (runtime via `api/`) consume the SAME `.md`. The negotiation endpoint is the runtime bridge that lets the Windows client (Epic 3, Story 3.2 "fetch the raw markdown via `Accept: text/markdown`") get the canonical source. Byte fidelity (AC6) matters because Markdig (the client's parser) must see the exact GFM source. [Source: architecture "content/ single source of truth"; epics.md Story 3.2 consume-side]
- **`Vary: Accept` correctness is a named architecture cross-cutting concern** ("Content-negotiation caching — `Vary: Accept` correctness"). The whole point of AC3 is that a shared cache (Azure CDN / browser cache / any intermediary) must key on `Accept` so a browser is NEVER served a stored markdown body. Emit `Vary: Accept` on BOTH branches. [Source: architecture Cross-Cutting Concerns #5]
- **Don't reinvent commodity plumbing (NFR-7):** use standard Azure Functions + standard SWA route/header config + standard HTTP content negotiation. No novel infrastructure. [Source: epics.md NFR-7]
- **Born-compatibility / SEO unaffected (NFR-4):** the browser HTML path stays first-class and crawlable — markdown is opt-in ONLY via an explicit `Accept: text/markdown`. A crawler sending `*/*` or `text/html` gets the same static HTML it gets today. Negotiation must NOT change what a browser/crawler receives. [Source: epics.md NFR-4; AC2]

and regenerate deterministically (`cd api && npm run build`). Chosen over B (emit into
`dist/` — higher `web/` regression surface + a second cacheable surface needing its own
`Vary`) and C (runtime fetch — network hop + fidelity drift risk).

**Pure handler / testability seam (AC8a).** `negotiate/negotiate.mjs` exports the pure
`negotiate({ slug, acceptHeader, readMd }) → { status, headers, body }`. RFC 9110
§12.5.1 Accept parsing: case-insensitive, OWS/`;param`-tolerant (charset ignored),
q-value aware (`q=0` = explicit reject), wildcards (`*/*`, `text/*`) NON-opt-in,
malformed → safe HTML default (never throws/400/500). Markdown wins only on a positive
`text/markdown` opt-in with q>0 that is not tied-or-beaten by an **explicit** HTML/
wildcard q (a bare `text/html` default does not outrank an explicit `text/markdown;q=0.9`).
`Vary: Accept` on BOTH branches (AC3). The Azure binding glue lives in
`negotiate/index.mjs` (v4 `app.http`) and the platform-agnostic request/response glue
in `negotiate/adapter.mjs` (no `@azure/functions` dep → CI-testable). The adapter
decodes the slug **exactly once** and rejects `..`/`%2F`/`%5C`/`%2E`/`%00`/double-encoded
(`%252F`)/absolute/UNC/drive-letter keys before the closed-map lookup; a hostile key
becomes the SAME uniform 404 as a missing page (no path leak, no oracle).

**SWA routing mechanism + trade-offs (AC7).** `web/public/staticwebapp.config.json`
(ships into `dist/`, the SWA-app-root location for a prebuilt upload) advertises
`Vary: Accept` on the page routes via per-route `headers` + `globalHeaders`, so the
**static host** itself emits `Vary: Accept` on the HTML branch — closing AC3 sub-bar (b):
a CDN/Front-Door edge can never cache an un-keyed HTML asset and replay it for a
`text/markdown` request. `Cache-Control` is left to inherit SWA defaults and is paired
with `Vary: Accept`, so no shared cache is more permissive than `Vary` protects.
SWA route rules cannot branch on a REQUEST `Accept` header, so the negotiation decision
necessarily lives inside the Function (the `negotiate()` handler); on the HTML branch the
Function 307-redirects to the canonical static page (`/<slug>`), keeping the browser path
on the fast static host. Trade-off: an `Accept`-keyed CDN rule isn't expressible purely
in `staticwebapp.config.json`, so the `Vary: Accept` correctness is enforced jointly by
the Function (both branches) AND the static-host headers — both surfaces advertise it.
`deploy-web.yml` now sets `api_location: 'api'` + installs api deps + runs the content
bundle step before packaging, without disturbing the prebuilt `web/dist` upload
(`skip_app_build: true`); `api/**` and `content/**` were added to `on.push.paths`. No
secret/subscription-id is hard-coded (the deploy token remains the only credential).

**Local-run instructions** are in `api/README.md`: `npm install && npm run build`, then
`npm test` (CI backstop, no emulator) and the gated `RUN_SWA_E2E=1 ... npm run test:e2e`
against `func start` / `swa start` (skipped in this CI — emulator impractical).
