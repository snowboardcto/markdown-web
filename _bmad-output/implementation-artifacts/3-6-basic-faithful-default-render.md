## Story

As a reader without a personality selected,
I want a faithful basic render by default,
so that the client is useful before any AI personalization exists.

## Context note (this is the SIXTH and FINAL `clients/windows/` feature story — the EPIC-3 FINISHER)

Stories 3.1–3.5 built and wired the client end-to-end; **3.6 is the visual default-theme pass + the "no personality selected → faithful basic render" default behavior + the consolidation/locking of the shell a11y (UX-DR9) as the Epic-3 exit criterion.** It is the last story of **Epic 3 (Read Markdown in the Native Client)**; Epic 4 then adds AI personalities on top of the seam this story pins.

### ⚠️ ENVIRONMENT CONSTRAINT — read before writing any code or test (drives AC6, Tasks for CI)

**This repo is developed on Linux with NO .NET SDK installed; WPF builds and runs ONLY on Windows.** The dev agent CANNOT build, run, or visually confirm anything locally. **Verification happens exclusively through `build-windows.yml` on `windows-latest`** (restore → build -c Release → test -c Release on `TheMarkdownWeb.sln`). The `paths:` filter (`clients/windows/**`) already covers every file this story touches. Therefore:

- Every acceptance bar that must be *checked* is checked **either by the compiler (build succeeds) or by an xUnit test that runs in `dotnet test` on `windows-latest`** — never "render the page and look at the spacing/colors."
- **`FlowDocument` + every flow element (`Paragraph`, `Run`, `Section`, `Table`, …) + `Window`/`FlowDocumentScrollViewer` are `DispatcherObject`s with STA thread affinity.** Any test that calls `Render(...)`, walks the FlowDocument tree, reads a `Paragraph.Margin`/`Run.Foreground`/`Section.Background`/`Table.BorderBrush`/`Block.LineHeight`, or constructs `MainWindow` MUST run on an STA thread via `Xunit.StaFact`'s `[StaFact]`. No `Window` is `.Show()`/`.ShowDialog()`'d; the runner is headless.
- **Prefer `[Fact]` for non-WPF logic.** The `RenderTheme` default value, the theme-option default values that are plain CLR (a brush is a WPF type → `[StaFact]`; an enum/double/Thickness-struct default may be `[Fact]` if it touches no `DispatcherObject`), and the Epic-4 override-seam contract are `[Fact]`-testable. Use `[StaFact]` for anything that constructs a `FlowDocument` element or reads a `Brush`/`DispatcherObject` property, and for the a11y `MainWindow` tests.

### Theming approach — PINNED (the contract for Step-4 TDD + Step-5 impl)

**Decision: extend `FlowDocumentRenderOptions` with the theme tokens + apply them in `FlowDocumentRenderer`'s existing mappers — NOT a separate post-render theme applier.** Rationale:
- The renderer already builds every element with its 3.3/3.4 markers; the cleanest additive change is to set the additional style properties (`Margin`/`LineHeight`/`Background`/`BorderBrush`/`Foreground`/`Padding`/`MarkerOffset`) on those same elements as they are built, reading the values from `_options`. A separate post-render walker would duplicate the tree-walking and risk drift.
- It keeps `Rendering` pure (style is set in-process, no I/O) and keeps the App wiring unchanged (the default `new FlowDocumentRenderer()` already flows through `ContentHostController`).
- The `RenderTheme Theme` property (default `Basic`) is the personality/Epic-4 seam: the renderer branches `if (_options.Theme == RenderTheme.Basic) ApplyBasicStyle(element)`. At 3.6 only the Basic branch exists; Epic 4 adds branches WITHOUT touching the Basic path.
- The theme brushes are `private static readonly` + `Freeze()`d (one instance per color, shareable across STA reconstructions — same pattern as the 3.4 palette so brush-by-reference comparisons are stable).

**Code-block background seam (the 3.4 deferral):** apply the `#f6f8fa` background at the code block's CONTAINER level, NOT on the per-token runs — otherwise the per-token `Foreground` brushes are untouched and `DistinctForegrounds(block).Count >= 2` (which counts run FOREGROUNDS, not backgrounds) still holds. Options (decide & document): set `Paragraph.Background` on the code `Paragraph` (simplest — the code block is already a `Paragraph`), OR wrap the code `Paragraph` in a `Section`/`BlockUIContainer` with the background + padding. **Default: `Paragraph.Background` + `Padding` on the existing code `Paragraph`** (no shape change → `Assert.Single(document.Blocks)` and the `BlockTag` helper in `CodeBlockTests` still see one `Paragraph` with `Tag==language`).
