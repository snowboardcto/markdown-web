# Story 5.1: Living Link

Status: review

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a reader,
I want a clear, one-click way to copy a `.md` page's canonical shareable URL — so that whoever I send it to opens the SAME page, rendered as HTML in their browser OR per their own personality in the native client,
so that sharing carries the per-reader "Living Link" magic without me having to know or hand-type the URL.

## Context note — this is the FIRST Epic-5 (post-MVP) story, and MOST of the AC is ALREADY SHIPPED

> **CRITICAL SCOPE FRAMING — read before writing any code.** The user chose "Run full sprint as-is" on a deliberately deferred post-MVP epic. The epic-level AC for 5.1 (epics.md lines 430–441) is: *a shared `.md` URL, opened in a browser → renders as HTML; opened in the native client → renders per the recipient's personality.* **Almost all of that behavior already exists from shipped MVP work and MUST be REUSED, not rebuilt:**
>
> - **Web HTML-at-a-URL** — Epic 2: every `content/**/*.md` is a static HTML page at `/<slug>` (`web/src/pages/[...slug].astro`, 157 E2E green). Story 2.7 made the page URL `/<slug>` **pure static HTML** for browsers (Option-2 decision: SWA can't branch on `Accept`, so raw markdown lives at the Function endpoint `/api/negotiate/<slug>`, NOT at the page URL). So **a browser opening `https://themarkdownweb.com/<slug>` already gets HTML — that half of the AC is DONE.**
> - **Native per-personality render** — Epic 4 (4.1–4.4 done): the reader picks a personality; `PersonalizationGateway` + `PersonalityEngine` re-render the page per reader (Basic default = faithful render; Cozy/Terminal/TL;DR/Plain/Translate/Audio). **A page rendered in the client already renders per the recipient's personality — that half of the AC is DONE.**
> - **Native fetch of a `.md` URL** — Epic 3 (3.1–3.6 done): the client's address bar takes a `.md` URL; `PageEndpointResolver.ToFetchEndpoint` maps an app-host `.md` URL → `/api/negotiate/<slug>` (via the byte-identical `SlugDeriver`), `MarkdownFetcher` GETs it with `Accept: text/markdown`, `NavigationController` renders it through the gateway. **A reader who PASTES a shared `.md` URL into the client and hits Enter already opens it, rendered per their personality — that path is DONE.**
>
> **Therefore the genuinely-NEW surface for 5.1 is the SHARE AFFORDANCE — a deterministic, one-click way to OBTAIN the canonical shareable URL of the page you're reading, on both surfaces, plus the canonical-URL plumbing that makes that link unambiguous.** It is NOT a new backend, NOT a persistence/link-shortener service, NOT accounts, NOT a new render path. The "Living Link" is just the page's own canonical `.md`-addressable URL, surfaced for copying. Everything that happens when the recipient OPENS it is already built.
>
> The scope of THIS story is intentionally lean: (1) a canonical absolute URL the web page declares for itself, (2) a "copy link" affordance on the web page that copies it, (3) a "copy/share link" affordance in the native client toolbar that copies the current page's canonical `.md` URL, and (4) tests proving the produced link is the canonical, round-trippable URL that opens correctly on both surfaces — reusing the EXISTING open/render paths, asserting they still work, and NOT regressing the 157 web specs or the WPF suite.

## Acceptance Criteria

> **The single epic-level AC (epics.md Story 5.1, lines 436–441):** *Given a shared `.md` URL, When it is opened in a browser Then it renders as HTML; And when opened in the native client, it renders per the recipient's personality (FR-15).* Because this is a broad outcome largely satisfied by shipped work, it is DECOMPOSED below into concrete, testable sub-ACs. AC1/AC2 lock the EXISTING "open → renders correctly" behavior as a regression contract (prove it still holds; do not rebuild it). AC3–AC5 are the genuinely-NEW share-affordance surface. AC6–AC8 are the standing scope/regression/CI guardrails.

1. **[Recipient opens the shared link in a BROWSER → HTML render (REUSE Epic 2 / 2.7; lock as regression contract)]** **Given** a canonical shareable page URL `https://themarkdownweb.com/<slug>` for a page that exists in the vault, **When** a browser opens it (a normal `Accept: text/html` / `*/*` request), **Then** it receives the **static server-rendered HTML page** (the exact behavior shipped in Epic 2 and preserved by Story 2.7's Option-2 decision — the page URL `/<slug>` is pure static HTML; markdown is opt-in ONLY at `/api/negotiate/<slug>` via `Accept: text/markdown`). This story **does NOT change** the page-URL HTML path, the Astro build, the content collection, the slug logic, or `staticwebapp.config.json`'s negotiation/`Vary` behavior. The proof is the EXISTING `web/` route + chrome behavior staying green (AC7) PLUS a focused assertion that the canonical URL a share affordance produces (AC3/AC4) resolves to a real HTML page for a representative vault page. *(AC1 — browser-open-renders-HTML; epics.md Story 5.1 line 439 "When it is opened in a browser Then it renders as HTML"; FR-5/FR-7; Story 2.7 Option-2 page-URL-stays-static-HTML decision in deferred-work.md.)*

2. **[Recipient opens the shared link in the NATIVE CLIENT → per-personality render (REUSE Epic 3 + Epic 4; lock as regression contract)]** **Given** the SAME canonical `.md` URL pasted into the native client's address bar (or otherwise navigated to), **When** the recipient opens it, **Then** the client fetches the raw markdown via the EXISTING path (`PageEndpointResolver.ToFetchEndpoint` → `/api/negotiate/<slug>`, `Accept: text/markdown`, `NavigationController` → `PersonalizationGateway`) and renders it **per the RECIPIENT's currently-selected personality** (Basic = faithful default on first run; Cozy/Terminal/TL;DR/Plain/Translate/Audio per Epic 4). This story **does NOT change** the fetch path, the slug derivation, the gateway, the engine, the personality selector, or the renderer. The proof is the EXISTING WPF open/render/personality behavior staying green (AC7) PLUS an assertion that a URL produced by the client's share affordance (AC5), when fed back through `PageEndpointResolver.ToFetchEndpoint`, maps to the SAME `/api/negotiate/<slug>` endpoint the original page fetched from (the link round-trips to the same content). *(AC2 — native-open-renders-per-personality; epics.md Story 5.1 lines 440–441 "when opened in the native client, it renders per the recipient's personality"; FR-9/FR-10/FR-14; Epic 3 fetch path + Epic 4 personalization, both done.)*

3. **[Canonical absolute URL: the web page declares its own shareable identity (NEW)]** **Given** a rendered web page at route `/<slug>`, **When** it is served, **Then** the page declares a single, deterministic **canonical absolute URL** `https://themarkdownweb.com/<slug>` so that the shared link is unambiguous regardless of how the page was reached (trailing slash, query string, the preview host, etc.). Concretely:
   - Configure Astro's `site` to the production origin (`https://themarkdownweb.com`) in `web/astro.config.mjs` (currently UNSET — `Astro.site` is `undefined` today, so there is no absolute-URL base). This is the ONLY `astro.config.mjs` change and it must not alter the existing build output for the 157 specs (the rehype plugins, Shiki theme, media copy, and route emission are untouched).
   - Add a `<link rel="canonical" href="<absolute-url>">` to the document `<head>` in `web/src/layouts/Page.astro` (currently the `<head>` has only charset/viewport/title — NO canonical link). The absolute URL is built from `Astro.site` + the page's route path, using `new URL(Astro.url.pathname, Astro.site)` (or the equivalent `Astro.url` resolved against `Astro.site`) so it derives from the SAME slug the route already emits — NO re-implementation of slug logic.
   - The canonical href is the page's own route (e.g. `/gear-guide` → `https://themarkdownweb.com/gear-guide`, `/sub/page` → `https://themarkdownweb.com/sub/page`, the index `/` → `https://themarkdownweb.com/`). It is well-formed, absolute, has NO trailing artifacts from the preview host or query string, and is identical for the content `[...slug]` route, the index, and (harmlessly) the 404/stub pages that share `Page.astro`.
   - The canonical link is **static HTML** (no `client:*`, no `<script>`) so it survives JS-disabled and is crawlable (NFR-4/FR-7 unaffected).
   - **Second-order watch (Second-Order Thinking) — setting `site` has a blast radius beyond the canonical link:** once `Astro.site` is defined, Astro begins emitting absolute URLs anywhere `Astro.site` is consumed (e.g. a future sitemap integration, RSS, or any component that resolves a relative path against `site`). For THIS story the ONLY consumer added is the canonical `<link>`, but verify the additive `site` key does not (a) change the dev/preview base path, (b) alter any already-emitted relative asset URL in `dist/` (media copy, CSS, Shiki), or (c) flip any existing `2-6-chrome.spec.ts` / build-output assertion. If `npm run build` output diff shows ONLY the new canonical `<link>` lines, the blast radius is contained — record this in the Dev Agent Record. Do NOT add a sitemap/RSS integration in this story (that would be a second-order scope expansion).

   Proven by Playwright `[spec]`s (a new `5-1-living-link.spec.ts`, mirroring `2-6-chrome.spec.ts`): for a representative vault page the served HTML contains exactly one `<link rel="canonical">` whose `href` is the absolute `https://themarkdownweb.com/<slug>` matching that page's route; the canonical href is present with JS disabled (the AC2-js-disabled pattern); the index page's canonical resolves to the site root. *(AC3 — canonical absolute URL; epics.md Story 5.1 "a shared `.md` URL"; the shared link must be canonical/unambiguous; `web/astro.config.mjs` has no `site`; `web/src/layouts/Page.astro` `<head>` has no canonical today.)*

4. **[Web share affordance: a "copy link" control that copies the canonical URL (NEW)]** **Given** a rendered web page, **When** the reader activates a **"copy link" affordance**, **Then** the page's **canonical absolute URL (AC3)** is copied to the clipboard, and the reader gets clear, non-blocking feedback (e.g. the control's label briefly confirms "Copied"). Concretely:
   - Add the affordance to the existing shared chrome so EVERY page inherits it with one edit — the natural home is the `site-header` (`web/src/components/SiteHeader.astro`, the sticky banner that already holds the wordmark + "the vision" + "Get the client") OR the end-of-page `pitch-card`; pick the header for reach and record the choice. It is a real, labeled, keyboard-reachable control (a `<button>`, NOT a bare glyph) with an accessible name (e.g. "Copy link"), meeting the NFR-6/UX-DR9 a11y floor (focusable, keyboard-activatable, AA contrast, an `aria-hidden` decorative glyph if any).
   - The copied value is the page's **canonical absolute URL** (AC3) — the SAME value declared in `<link rel="canonical">`, read from a single source (e.g. the canonical href in the DOM, or the same `Astro.site`-derived value passed to the component) so the copied link and the declared canonical can NEVER drift.
   - The clipboard write uses the standard `navigator.clipboard.writeText` with a graceful fallback / no-throw on older or permission-denied environments (the page must never crash; if the clipboard API is unavailable the control degrades — e.g. selects the text or shows the URL — rather than erroring). This is the ONLY interactive `<script>`/island this story adds to the web surface; it is small, scoped to the copy control, and does NOT affect the static content render (the page content stays server-rendered + crawlable + JS-disabled-safe; only the copy convenience needs JS, and the canonical link (AC3) is the JS-free shareable source of truth).
   - **Edge-case floor (Boundary & Edge Case Sweep) — the copy control must be TOTAL on the web side:** (i) **`navigator.clipboard` undefined** (insecure context / old browser) → no `TypeError`; degrade per fallback. (ii) **`writeText` returns a rejected Promise** (permission denied, document not focused) → the rejection is `.catch`'d; no unhandled rejection; feedback shows a non-success state or the fallback, never a thrown error. (iii) **JS disabled entirely** → the button may be inert BUT the canonical `<link>` (AC3) is still present as the JS-free shareable source, so the page is never left without a copyable URL (assert in the JS-disabled spec). (iv) **Double/rapid activation** → the "Copied" feedback reset does not stack or leak timers (a single in-flight reset). (v) **The control rendered on the index `/` and the 404/stub page** (which share the chrome) → it copies that page's own canonical (root / stub URL), never a stale or empty string.
   - **DECIDE-AND-DOCUMENT (record the pick + rationale in the Dev Agent Record):** (a) header vs pitch-card placement; (b) whether the copied value is read from the `<link rel="canonical">` href (single DOM source of truth) or injected as a component prop from `Astro.site`; (c) the exact "Copied" feedback mechanism and its reset timing; (d) the clipboard-unavailable fallback behavior.

   Proven by Playwright `[spec]`s in `5-1-living-link.spec.ts`: the "Copy link" control exists on a content page, is a `<button>` with a non-empty accessible name and is keyboard-focusable; activating it (using Playwright's clipboard permissions / `navigator.clipboard` read) yields the page's canonical absolute URL byte-equal to the page's `<link rel="canonical">` href; the success feedback appears. (Where Playwright clipboard read is constrained in CI, assert the control wires the canonical href as the copy source and the feedback toggles — document any gated assertion, mirroring the 2.7 emulator-gating precedent.) *(AC4 — web share affordance; UX-DR2/UX-DR3 chrome (the share control rides the existing header/pitch-card); UX-DR9/NFR-6 a11y; the "Living Link" is the page's canonical URL surfaced for one-click copy.)*

5. **[Native client share affordance: a toolbar "copy/share link" control that copies the current page's canonical `.md` URL (NEW)]** **Given** the native client has a page loaded, **When** the reader activates a **"copy link" toolbar control**, **Then** the **current page's canonical `.md` URL** is copied to the system clipboard, so the reader can paste a Living Link into any channel. Concretely:
   - The current page URL is read from the EXISTING `NavigationController.Current` (the `Uri` of the displayed page; `null` when nothing is loaded). Do NOT introduce a second source of truth for "the current page." When `Current` is `null` (nothing loaded) the control is a safe no-op / disabled — never a crash.
   - The copied value is the page's **canonical, shareable `.md` URL** for the canonical app host — i.e. the address-bar form a recipient can paste back into the client (and that, fed through `PageEndpointResolver.ToFetchEndpoint`, maps to the same `/api/negotiate/<slug>` the page was fetched from — AC2 round-trip). Define a small, pure, testable canonicalizer (e.g. `ShareLinkBuilder.ToShareUrl(Uri current) -> string`) that, for an app-host page URL (`PageEndpointResolver.IsAppHost` — REUSE it), produces the canonical `https://<host>/<path>` link; drop query/fragment consistently with how the endpoint resolver already drops them; preserve the host the reader is on (`themarkdownweb.com` / `www.themarkdownweb.com`). REUSE `SlugDeriver`/`PageEndpointResolver` logic so the share URL and the fetch endpoint can never disagree. For a non-app-host URL, return it unchanged (total, never throws).
   - The control is a **labeled, keyboard-reachable toolbar button** in `MainWindow.xaml` (a real `Button` with `AutomationProperties.Name` e.g. "Copy link", `Focusable`, a tab stop), placed in the toolbar WITHOUT displacing Back/Forward/Reload or breaking the address bar's `Width="*"` fill — add a NEW `Auto` toolbar `Grid` column (the established Q-Placement pattern from 4.2/4.4) and extend the tab order **sensibly** past the existing walk (Back 0 / Forward 1 / Reload 2 / AddressInput 3 / PersonalitySelector 4 / LanguagePicker 5 / ContentScroll 6) — give the new button a `TabIndex` that keeps the toolbar reachable before the content scroll (pick a value and re-number `ContentScroll` if needed, mirroring how 4.4 bumped `ContentScroll` 5→6; record the chosen integers).
   - The clipboard write uses WPF `System.Windows.Clipboard.SetText` (or an injected `IClipboard` seam for testability — preferred, so the copy can be unit-asserted without touching the real OS clipboard in CI), wrapped so a clipboard exception is caught and never throws into the UI (the WPF clipboard can transiently fail — total behavior).
   - **Edge-case floor (Boundary & Edge Case Sweep) — `ShareLinkBuilder.ToShareUrl` must be TOTAL and the round-trip must hold for every shape:** (i) **`null` input** → returns `null` (or empty) without throwing. (ii) **relative `Uri`** (`UriKind.Relative`) → total; no `InvalidOperationException` from touching `.Host`/`.Scheme`. (iii) **app-host URL with a trailing slash** (`/gear-guide/`) → canonicalizes to the SAME slug as the no-slash form (must not produce a divergent `/api/negotiate/<slug>` from the original fetch). (iv) **app-host URL with a query and/or fragment** (`?x=1#h`) → query+fragment dropped, identically to `PageEndpointResolver.ToFetchEndpoint`'s drop. (v) **the index/root URL** (`https://themarkdownweb.com/`) → a well-formed canonical, not an empty path. (vi) **`www.` vs apex host** → the reader's host is preserved (do NOT force one host) so the produced link round-trips through `IsAppHost` for BOTH. (vii) **percent-encoded path** (`/sub%2Fpage` or unicode slug) → decoded ONCE via the same path the resolver uses (REUSE; do not add a second decode and risk double-decode drift). (viii) **non-app-host URL** (`https://example.com/x`) → returned UNCHANGED, byte-for-byte (never silently rewritten to the app host).
   - **DECIDE-AND-DOCUMENT (Dev Agent Record):** (a) the exact canonical link form `ShareLinkBuilder` produces (scheme/host/path; query+fragment dropped) and why it round-trips through `PageEndpointResolver`; (b) the new toolbar column + `TabIndex` integers (+ any `ContentScroll` bump); (c) whether the clipboard write goes through an injected `IClipboard` seam (recommended for the CI `[Fact]`) or directly via `Clipboard.SetText` behind a try/catch; (d) the no-page-loaded behavior (disabled vs no-op).

   Proven by pure `[Fact]`s over `ShareLinkBuilder` (no window): an app-host page `Uri` (`https://themarkdownweb.com/gear-guide`, and with a trailing slash, a query, a fragment) → the canonical share URL `https://themarkdownweb.com/gear-guide`; the produced share URL fed back through `PageEndpointResolver.ToFetchEndpoint` equals the endpoint the original page fetched from (round-trip parity with `SlugDeriver`); a non-app-host URL is returned unchanged; `null`/relative input is total (no throw). Plus a `[Fact]` over the copy action with an in-memory `IClipboard` fake: with a loaded page the fake clipboard receives the canonical share URL; with no page loaded the action is a no-op and the clipboard is untouched, no throw. Plus a `[StaFact]` (mirroring `AddressBarWindowTests`/`PersonalitySelectorWindowTests`, construct-not-Show) asserting the named "copy link" `Button` exists in the toolbar, has a non-empty `AutomationProperties.Name`, is `Focusable` + a tab stop, and its `TabIndex` sits after `AddressInput` and the established controls and before `ContentScroll`. *(AC5 — native share affordance; UX-DR4/UX-DR5 toolbar; UX-DR9/NFR-6 a11y; REUSE `NavigationController.Current` + `PageEndpointResolver`/`SlugDeriver`; FR-15 "share a `.md` URL.")*

### Pre-mortem — how 5.1 ships broken (and the guard that prevents it)

> _Pre-mortem Analysis: assume it is post-merge and 5.1 is judged a failure. These are the failure modes, ranked, with the in-story guard that prevents each. No new scope — each guard is an existing AC/task assertion._

1. **Canonical drift — the copied link ≠ the declared canonical.** The web copy control reads a different value than `<link rel="canonical">` (e.g. it builds its own URL from `window.location` and picks up a trailing slash / query). → **Guard:** AC4 mandates a SINGLE source (read the canonical href from the DOM, or share one `Astro.site`-derived prop); Task 3 asserts byte-equality between the copied value and the `<link>` href. Decide-and-document the source.
2. **Slug divergence — the native share link does not round-trip.** `ShareLinkBuilder` re-implements slug/host logic instead of reusing `SlugDeriver`/`PageEndpointResolver`, so the share URL maps to a DIFFERENT `/api/negotiate/<slug>` than the page fetched from; the recipient opens the wrong/empty page. → **Guard:** AC5 mandates REUSE of `PageEndpointResolver.IsAppHost` + `SlugDeriver`; AC5/AC2 + Task 6 assert `ToFetchEndpoint(ToShareUrl(current)) == original endpoint` for every URL shape.
3. **Scope creep into 5.2 / backend.** A "link shortener," a "share count," a persistence row, or a Follow/Feed teaser sneaks in. → **Guard:** AC6 explicitly forbids new backend/persistence/accounts/Feed/render-path; the scope-boundary section enumerates the NOT-list; Task 8 DoD re-asserts scope held.
4. **Web regression — the 157 specs go red.** Setting `site` or adding the button to `SiteHeader.astro` changes build output / a chrome assertion broadly. → **Guard:** AC7 + the Second-order watch (AC3): the `site` key is the ONLY config change; if ONE `2-6-chrome.spec.ts` assertion legitimately shifts, reconcile that ONE minimally; verify the `dist/` diff is ONLY the canonical `<link>`.
5. **Crawlability/JS-disabled regression.** The shareable URL becomes JS-dependent (only obtainable via the copy script). → **Guard:** AC3 keeps the canonical `<link>` static in `<head>`; Task 3 asserts it is present with JS disabled — there is always a JS-free shareable source of truth.
6. **A throw reaches the UI.** `navigator.clipboard` undefined or WPF `Clipboard.SetText` transient `COMException` crashes the page/window; or `ShareLinkBuilder(null)` throws. → **Guard:** the AC4/AC5 edge-case floors mandate total/no-throw behavior; Task 3/Task 6 assert the unavailable-clipboard and `null`/relative paths.
