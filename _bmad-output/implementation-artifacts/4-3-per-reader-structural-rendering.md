### The CRITICAL constraint on what 4.3 can and cannot test (read before writing any test)

**The LLM is FAKED in CI — there is NO real model.** Therefore 4.3 **cannot** test the REAL structural QUALITY of AI output (whether Cozy *actually* reflows warmly, whether TL;DR *actually* compresses). That is a **runtime / manual** concern, validated on the reader's machine with a real key — **documented here, NOT unit-tested** (see "Runtime-only vs CI-provable" below and AC7).

What 4.3 **CAN** and DOES test deterministically in CI is the **PIPELINE CONTRACT** — the load-bearing, CI-provable surface:
- **(a) Prompt distinctness (AC2):** each structural persona has a **distinct, non-empty, well-formed** `SystemPrompt` that **encodes its transformation intent** (a markdown→markdown transform that PRESERVES valid markdown, with a DISTINCT structural emphasis: ordering / reading-level / emphasis / length). Pairwise-distinct; each mentions/encodes its intent (asserted by keyword presence).
- **(b) Routing (AC3):** the engine routes the **SELECTED** persona's **specific** `SystemPrompt` to `ILlmClient.CompleteAsync(systemPrompt, …)` — asserted via a **capturing fake** that records the `systemPrompt` it received: persona A's captured prompt `!=` B's, and each captured prompt `==` the registry's `persona.SystemPrompt`.
- **(c) Different output + visible change (AC4 + AC5):** with a fake LLM that maps **`systemPrompt → distinct output`**, two personas produce **DIFFERENT** `PersonalizationResult.Markdown` for the **same** source, the rendered `FlowDocument` differs, and **switching** persona re-renders in place to the different output (reuse the 4-2 `PersonalityRerenderCoordinator`). All deterministic; no real LLM.

### Q-Preference — what "changing a preference visibly changes the render" means at 4.3 (PINNED)

**RESOLVED: the PERSONA CHOICE ITSELF is the "preference" at 4.3. NO new reader-preference knob is added.**

- **Decision.** The epic's "And changing a preference visibly changes the render" is satisfied by the **persona selection** (already shipped by 4.2 as the `PersonalitySelectionViewModel` + `PersonalityRerenderCoordinator`). (4) Keeping the "preference" = the persona choice means 4.3 stays **prompt DATA + tests**, with minimal/no new production type — honoring "hold the line".

The existing `CountingLlmClient` (TestDoubles.cs) counts calls and returns ONE canned result for ALL prompts — it does NOT distinguish personas. 4.3 needs two small additions (TEST doubles only — never production):

  ```csharp
  internal sealed class CapturingLlmClient : ILlmClient {
      public string? LastSystemPrompt { get; private set; }
      public int Calls { get; private set; }
      public Task<LlmResult> CompleteAsync(string systemPrompt, string pageMarkdown, ReaderContext ctx, CancellationToken ct) {
          Calls++; LastSystemPrompt = systemPrompt;
          return Task.FromResult(LlmResult.Success("# captured"));
      }
  }
  ```
- **systemPrompt-keyed fake (AC4/AC5) — PINNED form (a), Q-Marker resolved:** returns a deterministic output that is a **pure function of the received `systemPrompt`** (a prompt-derived marker prepended to the body), NOT an exact-prompt→canned map. The marker must satisfy `marker(a) != marker(b)` whenever `a != b` (the four real prompts are pairwise-distinct by AC2, so this holds):
  ```csharp
  internal sealed class KeyedLlmClient : ILlmClient {
      public Task<LlmResult> CompleteAsync(string systemPrompt, string pageMarkdown, ReaderContext ctx, CancellationToken ct)
          => Task.FromResult(LlmResult.Success($"{Marker(systemPrompt)}\n\n{pageMarkdown}"));
      // Marker: a STABLE pure token of the prompt — e.g. a short hex digest, or "persona-" + systemPrompt.Length.
      // NOT a literal of any specific prompt's wording (so the test survives later prompt-wording refinement).
      internal static string Marker(string systemPrompt) =>
          "persona-" + Convert.ToHexString(SHA256.HashData(Encoding.UTF8.GetBytes(systemPrompt)))[..8];
  }
  ```

## Q-Marker — the keyed-fake output shape (RESOLVED — was the lone open question)

**RESOLVED: the keyed fake derives its output DETERMINISTICALLY FROM the received `systemPrompt` (form (a) — a prompt-derived marker). No exact-prompt→canned-map (form (b) is rejected as brittle).**

- **Decision (the pinned `KeyedLlmClient` contract — Step-4/Step-5 BIND to this exact shape):**
  - The fake returns `LlmResult.Success("<persona-marker>\n\n" + pageMarkdown)` where `<persona-marker>` is a **stable, pure function of the received `systemPrompt`** — NOT a literal of any specific prompt's wording. The recommended marker is a short stable token of the prompt, e.g. a hex prefix of a `SHA256`/`GetHashCode`-style digest, or `"persona-" + systemPrompt.Length`. The ONLY requirement: **`marker(promptA) != marker(promptB)` whenever `promptA != promptB`** (an injective-enough digest over the four real, pairwise-distinct prompts — guaranteed by AC2's pairwise-distinctness).
  - Because the engine forwards `persona.SystemPrompt` verbatim (`PersonalityEngine.cs` line 82) and the four real prompts are pairwise-distinct (AC2), `cozy.SystemPrompt != terminal.SystemPrompt` ⇒ `marker(cozy) != marker(terminal)` ⇒ **`cozyResult.Markdown != terminalResult.Markdown`** — deterministically, for the SAME source, with NO real model.
- **What the AC4/AC5 tests assert (pinned, ROBUST — no hardcoded prompt wording):**
  - `cozyResult.Markdown != terminalResult.Markdown` (the two outputs differ — the load-bearing assertion). This holds **without** the test referencing any exact prompt string, so it survives later prompt-wording refinement.
  - At the render level: the cozy marker appears in the cozy `FlowDocument` text and NOT in the terminal one, and vice-versa (`markerOf(cozyPrompt)` ∈ cozyRender, ∉ terminalRender) — derived from the registry prompt via the SAME marker function the fake uses, so still no literal-prompt coupling.
  - The fake also **prepends, never replaces** the body (`marker + "\n\n" + pageMarkdown`), so the output is valid renderable markdown and the source is preserved (the difference is a structural prefix, not a destroyed document).
- **Why form (b) is rejected:** mapping exact registry prompts → canned markdown couples every AC4/AC5 test to the precise prompt wording; any AC2-permitted wording refinement would silently break the difference proof. Form (a) is robust to wording changes while still deterministic and CI-safe.
- **Capturing fake (AC3) is the complementary half** and is also pinned (see Dev Notes "The capturing / keyed fake"): it records `LastSystemPrompt` + `Calls` verbatim and returns a fixed canned success — it does NOT key its output (its job is to prove the EXACT forwarded prompt, asserted byte-equal against `PersonaRegistry.Seed`'s prompt, not to differ output). Routing (AC3, capturing fake) and different-output (AC4/AC5, keyed fake) are deliberately two distinct fakes with two distinct jobs.
