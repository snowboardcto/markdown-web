## Story

As a reader,
I want fenced code blocks highlighted,
so that code is readable and the render feels like GitHub.

## Context note (this is the FOURTH `clients/windows/` feature story — Epic 3; it COLORS the bedrock fenced-code render from 3.3)

Story 3.1 shipped the **app shell**; Story 3.2 shipped the **`.md`-only address bar + fetch**; **Story 3.3** shipped the **bedrock render** — the pure `Rendering` library turns a markdown string into a WPF **`FlowDocument`** via Markdig (`FlowDocumentRenderer.Render(string)`), mapping every GFM element family. Crucially, **3.3 deliberately rendered fenced code as a SINGLE-color monospace block** (AC5: `DistinctForegrounds(block).Count <= 1`, "no colors yet") and **recorded the fence's language info-string in `Paragraph.Tag`** (e.g. `"csharp"`) precisely so this story can colorize it. All three prior stories are `done` and green on `windows-latest` CI (3.3 run #16).

**Story 3.4 is the syntax-highlighting layer:** a fenced code block with a recognized language gets **GitHub-style token coloring** (keywords, strings, comments, numbers, functions in distinct colors) by tokenizing the code with **ColorCode** and emitting one WPF `Run` per token with the matching `Foreground` brush — approximating the web's Shiki **github-light** theme. It **inverts** 3.3's "single foreground" proof: a known-language block now carries **multiple** distinct foreground brushes (the provable evidence that highlighting happened). An **unknown or missing language** degrades gracefully to the 3.3 plain single-color mono block — **never an error**.

**This story hooks into EXACTLY 3.3's fenced-code path.** In `FlowDocumentRenderer`, `MapBlock` routes `FencedCodeBlock`/`CodeBlock` to `MapCodeBlock(code, info)`, which today builds a mono `Paragraph` tagged with the language (first word of the info-string) holding one un-`Foreground`-set `Run` per source line separated by `LineBreak`. Story 3.4 changes ONLY the inside of that path: when the language resolves to a ColorCode `ILanguage`, the per-line runs are replaced by **colored token runs**; the block's outer contract (top-level block, `Tag` == language, mono `FontFamily`, verbatim text incl. newlines) is **preserved unchanged** so 3.3's other CodeBlockTests asserts and downstream stories (3.5/3.6) keep working.

**Story 3.4 does NOT:** add line numbers, a copy button, themes/theme-switching, or per-personality code styling (Epic 4); touch in-client links/media/navigation (3.5); do the basic-faithful polish pass (3.6 — exact code-block background/`6px` radius/measure stays deferred); change ANY non-code rendering (headings/emphasis/paragraphs/lists/task-lists/blockquotes/tables/images are all 3.3 and stay byte-for-byte as-is); or highlight **inline** code (`` `x` `` stays plain mono — GitHub does not color inline code).

## Acceptance Criteria

1. **[ColorCode integration: language → tokens → colored WPF `Run`s]** **Given** the pure `Rendering` library and a fenced code block whose info-string names a language ColorCode recognizes, **When** `FlowDocumentRenderer.Render(markdown)` maps that block, **Then** the renderer **tokenizes the code with ColorCode** and emits the code block's text as a sequence of `Run`s where each run's `Foreground` is the brush mapped from that token's ColorCode scope. Concretely:
   - The renderer references **`ColorCode.Core`** (the tokenizer-only ColorCode-Universal package — Parser + `Languages.FindById` + `CodeColorizerBase` + `StyleDictionary` + `Scope`; pure managed, no net/AI/webview — see Dev Notes for the pinned version + fallback).
   - A new internal type (e.g. `clients/windows/Rendering/Highlighting/FlowDocumentCodeColorizer.cs`) subclasses ColorCode's `public abstract CodeColorizerBase` and overrides `protected override void Write(string parsedSourceCode, IList<Scope> scopes)` to produce WPF `Run`s (this is the same pattern ColorCode's own UWP `RichTextBlockFormatter` uses; net10 has no shipped WPF formatter so we write this small one). It exposes a public entry method (e.g. `IReadOnlyList<Run> GetRuns(string code, ILanguage language)`) that drives the base via **`languageParser.Parse(code, language, (parsedSourceCode, captures) => Write(parsedSourceCode, captures))`** (the `languageParser` field comes from `CodeColorizerBase`'s ctor; pass `null` styles/parser to take the defaults), accumulating the produced `Run`s into a field the entry method returns. Each emitted `Run` uses the configured **monospace `FontFamily`** and a `Foreground` from the scope→brush map (AC2). Text between/outside scopes (whitespace, punctuation, unscoped text) is emitted as default-foreground mono runs; the **full code text is preserved verbatim** (concatenating all runs' `Text`, including newlines, reproduces the source — the `Write` walk covers the gaps between scope offsets, exactly as `RichTextBlockFormatter` does).
   - The language is resolved from the fence info-string's first word via `Languages.FindById(language)` (case-insensitive; e.g. `csharp`, `cs`/`c#`, `js`/`javascript`, `ts`/`typescript`, `json`, `python`/`py`, `html`, `xml`, `css`, `bash`/`shell`, etc. — whatever ColorCode supports out of the box, plus a small alias map for the common GFM fence ids ColorCode doesn't natively id, see Dev Notes).
   - Highlighting is **on by default**. `FlowDocumentRenderOptions` gains `public bool SyntaxHighlighting { get; init; } = true;`. When `false`, `MapCodeBlock` produces the 3.3 single-color block (escape hatch; default path is highlighted). No separate ctor or public method is added — highlighting plugs into the existing `Render` automatically via `MapCodeBlock`.
   - The block's **3.3 outer contract is preserved**: it is still a top-level block, still carries `Tag` == the language string (or null/empty when absent), still uses the mono `FontFamily`, and still preserves the code verbatim incl. line breaks.

   Proven by a `[StaFact]`: render a `` ```csharp\nvar x = 1;\n// note\n``` `` block; assert one top-level code block, `Tag == "csharp"`, mono `FontFamily` on every run, and the concatenated run text equals the verbatim code (incl. the newline). *(AC1 — epics.md Story 3.4 line 335 "tokens are syntax-highlighted via ColorCode"; architecture.md line 102 "ColorCode-Universal (.NET → FlowDocument runs)", line 133 Rendering/Highlighting/ ColorCode.)*

2. **[GitHub-light-ish token palette]** **Given** a highlighted code block, **When** scopes are mapped to brushes, **Then** the renderer uses a **deterministic, total scope→`SolidColorBrush` map approximating GitHub github-light** (the DESIGN.md `colors.code` palette, matching the web's Shiki github-light look), with these **exact ARGB** values (all full-opacity, alpha `0xFF` — `Color.FromRgb(r,g,b)` yields `#FF……`):

   | Token kind | Hex | ARGB (`A,R,G,B`) | ColorCode `ScopeName` constants mapped |
   |---|---|---|---|
   | keyword | `#cf222e` | `255, 207, 34, 46` | `Keyword`, `ControlKeyword`, `PreprocessorKeyword` |
   | string | `#0a3069` | `255, 10, 48, 105` | `String`, `StringCSharpVerbatim`, `StringEscape` |
   | comment | `#59636e` | `255, 89, 99, 110` | `Comment`, `XmlComment`, `XmlDocComment`, `XmlDocTag` |
   | number | `#0550ae` | `255, 5, 80, 174` | `Number` |
   | function/type | `#8250df` | `255, 130, 80, 223` | `ClassName`, `HtmlElementName`, `XmlAttribute`, `XmlAttributeQuotes` (type/identifier-ish) |
   | **default (fallback)** | `#1f2328` | `255, 31, 35, 40` | **ANY other / unmapped / unscoped `ScopeName`** |

   The map is **total** — any ColorCode scope name that isn't a key maps to the **default** brush `#1f2328` (never throws, never null). The map is a `private static readonly` (a `Dictionary<string, SolidColorBrush>` keyed on the `ScopeName` string constants) **constructed once**, and every brush is **`Freeze()`d** at construction so it is shareable across STA reconstructions and `DistinctForegrounds` counts the SAME brush instance by reference (one brush per color, not a fresh allocation per token). (Exact code-block **background** fill `#f6f8fa` / `6px` radius is Story 3.6 polish — NOT required here; this story colors the **token foregrounds** only.)

   Proven by a `[StaFact]`: render a C# block containing a keyword, a string literal, and a comment; locate the run whose `Text` is the keyword and assert its `Foreground` is a `SolidColorBrush` whose `.Color == Color.FromRgb(0xCF, 0x22, 0x2E)` (ARGB `255,207,34,46`), the string run's `== Color.FromRgb(0x0A, 0x30, 0x69)`, and the comment run's `== Color.FromRgb(0x59, 0x63, 0x6E)` (reads the EXACT produced ARGB; proves the palette is applied, not just "some color"). *(AC2 — DESIGN.md `colors.code` keyword/string/comment/function/number + `colors.fg`; "Code syntax palette … GitHub-light-aligned", DESIGN.md line 58.)*
