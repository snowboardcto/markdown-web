# Story 2.6: Site header and pitch card

Status: in-progress

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a first-time visitor,
I want every rendered web page to convey the vision and how to get the native client,
so that the web recruits me to the native experience instead of leaving me on one fixed view.

## Acceptance Criteria

1. **Given** ANY rendered web page (every content page emitted by `[...slug].astro`, the index at `/`, AND the 404 page), **When** it displays, **Then** a **sticky site-header** is present at the top of the page showing: (a) the `.md the markdown web` wordmark — a small monospace `.md` chip in `{colors.ink}` beside the lowercase sans wordmark text "the markdown web"; (b) a **"the vision"** link; and (c) a **"Get the client"** CTA (solid `{colors.ink}` pill button, `{rounded.button}` 9px). The header is `position: sticky; top: 0`, translucent with a blur (`rgba(255,255,255,.85)` + `backdrop-filter: blur`), with a bottom hairline (`{colors.border}`). It is real **page content** rendered as a semantic `<header>` element — NOT fake browser chrome (DESIGN.md "Don't add fake browser chrome to the web"). Because the chrome lives in the shared `Page.astro` layout, it appears on EVERY surface with one change. **Accessibility (NFR-6/UX-DR9):** the header is a top-level landmark (a bare `<header>` that is a direct child of `<body>` exposes the `banner` role implicitly — do NOT nest it inside `<main>`/`<article>`, which would strip the banner role); the two interactive controls ("the vision", "Get the client") are real `<a href>` elements with **accessible names equal to their visible text** (if the CTA carries the decorative `▣` glyph, it must NOT pollute the accessible name — either keep `▣` as a plain text sibling of the label or wrap it in `aria-hidden="true"`, so the link's accessible name stays exactly "Get the client", not "▣ Get the client"); the CTA pill meets **WCAG AA contrast** — white (`#ffffff`) text on `{colors.ink}` `#0d1117` is ~18.4:1, well past the 4.5:1 floor (do NOT lighten the pill below AA); the sticky bar must not trap or hide keyboard focus — tab order is wordmark→"the vision"→"Get the client"→(into `<main>`), and because `position: sticky` keeps the bar in normal flow it does NOT overlap the `<main>` content or obscure a focused in-content element (no negative-margin/`fixed` overlay that would cover focused content beneath it). *(AC1 — sticky site-header with wordmark + "the vision" + "Get the client" CTA on every page, semantic `banner` landmark, AA-contrast ink CTA pill, accessible link names, keyboard-reachable with no focus overlap; UX-DR2/UX-DR3/UX-DR9, NFR-6, DESIGN.md `site-header`/`get-client-cta`, epics.md Story 2.6)*

2. **Given** any rendered web page, **When** the reader reaches the foot of the page, **Then** an **end-of-page pitch-card** is present (a semantic element AFTER the main content, e.g. a `<section>` / `<footer>` region — NOT inside `<article>`'s reading flow) showing: (a) the **vision headline** "You're reading one fixed view. There's a better one."; (b) the **vision body** copy (per EXPERIENCE.md Voice and Tone — see AC4); (c) a **"Get the Markdown Web client"** CTA; and (d) a **"Why a markdown web?"** link. It is a subtly-elevated gradient surface, `{rounded.card}` (14px), at the reading measure (`{typography.measure}` 760px, centered). The pitch-card is the page's built-in "upgrade moment" (EXPERIENCE.md KF-1). **Accessibility (NFR-6/UX-DR9):** the pitch-card lives OUTSIDE `<main>` (direct child of `<body>`, after `</main>`) so its heading does not perturb the in-content heading outline; render it as a single `<footer>` (implicit `contentinfo` landmark when a direct child of `<body>`) OR a `<section>` — but if `<section>`, give it an accessible name (`aria-label` or an `aria-labelledby` pointing at the pitch headline) so it is a NAMED region rather than an anonymous one; the two interactive controls are real `<a href>` with accessible names equal to their visible text (the decorative `▣` / `→` glyphs must be `aria-hidden` or kept out of the accessible name, so the names stay exactly "Get the Markdown Web client" and "Why a markdown web?"); the pitch CTA pill reuses the same white-on-`{colors.ink}` AA-passing contrast as the header CTA, and the "Why a markdown web?" `{colors.link}` `#0969da` link text on the gradient surface (`#fbfdff→#f3f7fc`) stays ≥4.5:1 (link `#0969da` on `#fff` is ~4.6:1; the near-white gradient does not drop it below AA). *(AC2 — end-of-page pitch-card: headline + body + "Get the Markdown Web client" + "Why a markdown web?" on every page, semantic named region outside `<main>`, AA-contrast CTA + link, accessible link names; UX-DR2/UX-DR9/UX-DR10, NFR-6, DESIGN.md `pitch-card`, epics.md Story 2.6)*

3. **Given** the chrome must apply uniformly to every web surface, **When** the site builds, **Then** the site-header + pitch-card are added to the **shared `web/src/layouts/Page.astro` layout** (NOT duplicated per page), so all three surfaces inherit them: content pages (`[...slug].astro` → `Page`), the vault index (`index.astro` → `Page`), and the 404 page (`404.astro` → `Page`). No surface renders its own `<html>` outside `Page` (verified: all three already wrap their body in `<Page>`). The header sits BEFORE `<main>`; the pitch-card sits AFTER `<main>`, both as direct children of `<body>` (the wordmark/CTA are page chrome, the content `<h1>` stays the sole heading inside `<article>`). *(AC3 — chrome lives in the shared `Page.astro` layout; applies to content + index + 404 with one change; epics.md Story 2.6, architecture browser path)*

4. **Given** microcopy must match EXPERIENCE.md "Voice and Tone" (plain, confident, developer-credible, never salesy), **When** the chrome renders, **Then** the copy matches the EXPERIENCE.md anchors EXACTLY — verbatim, including apostrophe, em-dash, and `.md`/code styling:
   - Pitch headline (EXPERIENCE.md line 34, verbatim): **`You're reading one fixed view. There's a better one.`** — a curly **`'`** (U+2019) in both `You're` and `There's` is acceptable (EXPERIENCE.md/mockup use the typographic apostrophe; if you emit a straight `'`, keep it consistent and assert the form you ship). Two sentences, one period each, NO exclamation mark.
   - Pitch body (EXPERIENCE.md line 35 anchor): the load-bearing tail **MUST appear verbatim** — `…your own AI personality renders every` `.md` `page your way — your layout, your language, your reading level. Same file. Your shape.` — note the **em-dash `—`** (U+2014, not a hyphen) before "your layout", the `.md` rendered as inline `<code>` (mockup line 98), and the final cadence **`Same file. Your shape.`** as two terse sentences. The web-v2 mockup lead-in framing *"This page is plain markdown, rendered the same for everyone. In the **Markdown Web client**, …"* MAY be used verbatim (the `Markdown Web client` is bolded `<strong>` in the mockup, line 97); the lead-in is allowed but the EXPERIENCE.md tail from "your own AI personality" through "Your shape." is the non-negotiable anchor.
   - Header wordmark text (DESIGN.md line 50 + Brand & Style): **`.md the markdown web`** — the mono `.md` chip (an own `<span>` so the chip is `.md` and the rest is the lowercase sans " the markdown web"); accessible/visible text reads `.md the markdown web`.
   - Header links/CTA labels (epics.md Story 2.6 line 262): **`the vision`** (lowercase, no trailing punctuation), **`Get the client`** (the mockup prefixes a decorative `▣`; the load-bearing, accessible-name string is exactly `Get the client` — see AC1 on keeping `▣` out of the accessible name).
   - Pitch-card CTA + link labels (epics.md Story 2.6 line 263): **`Get the Markdown Web client`** (note: the header CTA is the SHORT `Get the client`; the pitch CTA is the LONGER `Get the Markdown Web client` — these two are DELIBERATELY different strings, do NOT unify them), **`Why a markdown web?`** (the mockup renders it `Why a markdown web? →`; the `→` is decoration, the load-bearing string is `Why a markdown web?` with the question mark).
   No invented marketing copy; no exclamation-driven sales language; no "Sign up" / "Free" / "Download now" embellishment not in the anchors. *(AC4 — microcopy matches EXPERIENCE.md Voice and Tone verbatim for the anchored strings, header-vs-pitch CTA strings kept distinct, decorative glyphs excluded from accessible names; epics.md "the microcopy matches EXPERIENCE.md Voice and Tone", UX-DR10)*

Both "the vision" and "Why a markdown web?" MAY point at the same `/vision` target. Pick ONE, record it.
   Whatever is chosen, the targets must NOT break the build and MUST be documented in the Dev Agent Record as Epic-3 / future-content stubs. *(AC5 — "Get the client" and "the vision"/"Why a markdown web?" links resolve to documented stub targets (client download = Epic 3, vision = future content); no dangling/404 link, no build break; epics.md FR-9 handoff, EXPERIENCE.md get-client-cta/pitch-card "link to the client download")*

6. **Given** the chrome must preserve the established reading experience, **When** any page renders, **Then** it stays: **themed** (Story 2.2 `github.css` tokens + reading measure — the chrome reuses the SAME design tokens, light default), **server-rendered & JS-free** (NO `client:*` island, NO runtime `<script>` — the header/CTA/pitch are static HTML+CSS only), **crawlable** (valid static HTML a crawler parses; the links are plain `<a href>`), and structurally clean: a SINGLE `<html lang>` → `<head>` → `<body>` shell with **exactly ONE `<h1>` per page** — **the header wordmark MUST NOT be an `<h1>`** (render it as a `<span>`/`<a>`, per the web-v2 mockup) and the **pitch-card headline MUST NOT be an `<h1>`** (the mockup uses `<h4>`/a non-h1 heading or a styled `<p>`/`<strong>`), so the page's existing content `<h1>` (inside `<article>`) remains the sole `<h1>`. Header/footer regions are semantic (`<header>` for the site-header; the pitch-card may be a `<footer>`/`<section>`). *(AC6 — themed, JS-free, crawlable, single-`<h1>` preserved (wordmark/pitch headline are NOT h1); FR-5/FR-7/NFR-4, UX-DR1/UX-DR9, DESIGN.md light-default + reuse-tokens)*
