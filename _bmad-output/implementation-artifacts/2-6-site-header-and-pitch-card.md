# Story 2.6: Site header and pitch card

Status: ready-for-dev

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a first-time visitor,
I want every rendered web page to convey the vision and how to get the native client,
so that the web recruits me to the native experience instead of leaving me on one fixed view.

## Acceptance Criteria

1. **Given** ANY rendered web page (every content page emitted by `[...slug].astro`, the index at `/`, AND the 404 page), **When** it displays, **Then** a **sticky site-header** is present at the top of the page showing: (a) the `.md the markdown web` wordmark — a small monospace `.md` chip in `{colors.ink}` beside the lowercase sans wordmark text "the markdown web"; (b) a **"the vision"** link; and (c) a **"Get the client"** CTA (solid `{colors.ink}` pill button, `{rounded.button}` 9px). The header is `position: sticky; top: 0`, translucent with a blur (`rgba(255,255,255,.85)` + `backdrop-filter: blur`), with a bottom hairline (`{colors.border}`). It is real **page content** rendered as a semantic `<header>` element — NOT fake browser chrome (DESIGN.md "Don't add fake browser chrome to the web"). Because the chrome lives in the shared `Page.astro` layout, it appears on EVERY surface with one change. *(AC1 — sticky site-header with wordmark + "the vision" + "Get the client" CTA on every page; UX-DR2/UX-DR3, DESIGN.md `site-header`/`get-client-cta`, epics.md Story 2.6)*

2. **Given** any rendered web page, **When** the reader reaches the foot of the page, **Then** an **end-of-page pitch-card** is present (a semantic element AFTER the main content, e.g. a `<section>` / `<footer>` region — NOT inside `<article>`'s reading flow) showing: (a) the **vision headline** "You're reading one fixed view. There's a better one."; (b) the **vision body** copy (per EXPERIENCE.md Voice and Tone — see AC4); (c) a **"Get the Markdown Web client"** CTA; and (d) a **"Why a markdown web?"** link. It is a subtly-elevated gradient surface, `{rounded.card}` (14px), at the reading measure (`{typography.measure}` 760px, centered). The pitch-card is the page's built-in "upgrade moment" (EXPERIENCE.md KF-1). *(AC2 — end-of-page pitch-card: headline + body + "Get the Markdown Web client" + "Why a markdown web?" on every page; UX-DR2/UX-DR10, DESIGN.md `pitch-card`, epics.md Story 2.6)*

3. **Given** the chrome must apply uniformly to every web surface, **When** the site builds, **Then** the site-header + pitch-card are added to the **shared `web/src/layouts/Page.astro` layout** (NOT duplicated per page), so all three surfaces inherit them: content pages (`[...slug].astro` → `Page`), the vault index (`index.astro` → `Page`), and the 404 page (`404.astro` → `Page`). No surface renders its own `<html>` outside `Page` (verified: all three already wrap their body in `<Page>`). The header sits BEFORE `<main>`; the pitch-card sits AFTER `<main>`, both as direct children of `<body>` (the wordmark/CTA are page chrome, the content `<h1>` stays the sole heading inside `<article>`). *(AC3 — chrome lives in the shared `Page.astro` layout; applies to content + index + 404 with one change; epics.md Story 2.6, architecture browser path)*

4. **Given** microcopy must match EXPERIENCE.md "Voice and Tone" (plain, confident, developer-credible, never salesy), **When** the chrome renders, **Then** the copy matches the EXPERIENCE.md anchors EXACTLY:
   - Pitch headline: **"You're reading one fixed view. There's a better one."** (EXPERIENCE.md line: Pitch headline anchor)
   - Pitch body conveys: **"…your own AI personality renders every `.md` page your way — your layout, your language, your reading level. Same file. Your shape."** (EXPERIENCE.md Pitch body anchor; the web-v2 mockup frames it as: *"This page is plain markdown, rendered the same for everyone. In the **Markdown Web client**, your own AI personality renders every `.md` page your way — your layout, your language, your reading level. Same file. Your shape."* — the bolded final sentences are the load-bearing anchor and MUST appear verbatim; the lead-in framing may match the mockup).
   - Header wordmark text: **`.md the markdown web`** (mono `.md` chip + "the markdown web").
   - Header links/CTA labels: **"the vision"**, **"Get the client"**.
   - Pitch-card CTA + link labels: **"Get the Markdown Web client"**, **"Why a markdown web?"**.
   No invented marketing copy; no exclamation-driven sales language. *(AC4 — microcopy matches EXPERIENCE.md Voice and Tone verbatim for the anchored strings; epics.md "the microcopy matches EXPERIENCE.md Voice and Tone")*

5. **Given** the link targets for "the vision", "Get the client", "Get the Markdown Web client", and "Why a markdown web?", **When** the chrome renders, **Then** each link/CTA points at a **documented, deterministic target** — and since the native client download (Epic 3) and a vision/manifesto content page do NOT exist yet, the targets are **explicit, documented stubs** that do NOT 404 the build or dangle:
   - **"Get the client" / "Get the Markdown Web client"** → a single shared **client-download stub target** (Decision A — RECOMMENDED: a stable in-repo route `/get`, served by a new minimal `web/src/pages/get.astro` placeholder page rendered through `Page` so the link resolves to a real 200 page that says the client is coming in Epic 3; ALTERNATIVELY a bare `#` anchor if you choose not to add a route). Pick ONE, record it, and make tests assert the ACTUAL chosen target.
   - **"the vision" / "Why a markdown web?"** → a **vision/manifesto stub target** (Decision B — RECOMMENDED: a stable route `/vision` served by a new minimal `web/src/pages/vision.astro` (or a `content/vision.md` page — but note that would then appear in the 2.5 index listing and change the index one-to-one count, so a non-collection `web/src/pages/vision.astro` is the lower-blast-radius choice); ALTERNATIVELY `#`). Both "the vision" and "Why a markdown web?" MAY point at the same `/vision` target. Pick ONE, record it.
   Whatever is chosen, the targets must NOT break the build and MUST be documented in the Dev Agent Record as Epic-3 / future-content stubs. *(AC5 — "Get the client" and "the vision"/"Why a markdown web?" links resolve to documented stub targets (client download = Epic 3, vision = future content); no dangling/404 link, no build break; epics.md FR-9 handoff, EXPERIENCE.md get-client-cta/pitch-card "link to the client download")*

6. **Given** the chrome must preserve the established reading experience, **When** any page renders, **Then** it stays: **themed** (Story 2.2 `github.css` tokens + reading measure — the chrome reuses the SAME design tokens, light default), **server-rendered & JS-free** (NO `client:*` island, NO runtime `<script>` — the header/CTA/pitch are static HTML+CSS only), **crawlable** (valid static HTML a crawler parses; the links are plain `<a href>`), and structurally clean: a SINGLE `<html lang>` → `<head>` → `<body>` shell with **exactly ONE `<h1>` per page** — **the header wordmark MUST NOT be an `<h1>`** (render it as a `<span>`/`<a>`, per the web-v2 mockup) and the **pitch-card headline MUST NOT be an `<h1>`** (the mockup uses `<h4>`/a non-h1 heading or a styled `<p>`/`<strong>`), so the page's existing content `<h1>` (inside `<article>`) remains the sole `<h1>`. Header/footer regions are semantic (`<header>` for the site-header; the pitch-card may be a `<footer>`/`<section>`). *(AC6 — themed, JS-free, crawlable, single-`<h1>` preserved (wordmark/pitch headline are NOT h1); FR-5/FR-7/NFR-4, UX-DR1/UX-DR9, DESIGN.md light-default + reuse-tokens)*
